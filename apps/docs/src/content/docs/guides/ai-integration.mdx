`@anvilkit/plugin-ai-copilot` is deliberately headless: it does not
ship a provider, does not bundle an API client, and does not know
what model you run. Instead, it accepts a single host-owned callback:

```ts
createAiCopilotPlugin({
	generatePage: async (prompt, ctx) => {
		// you choose the LLM, you hold the secrets
		return somePageIR;
	},
	puckConfig,
});
```

This guide explains the trust model that shape choice enforces, the
exact `generatePage()` contract, how the schema context is built, the
validator boundary every response crosses before dispatch, how to
surface errors, cost-control knobs, and three worked adapters against
real providers. It ends with an explicit "do not do this" section.

## 1. Trust model

Three principals sit between a user's prompt and a rendered page:

```
┌─────────────────┐    ┌──────────────┐    ┌──────────────┐
│   Studio UI     │───▶│   your host  │───▶│    LLM       │
│ (this package)  │    │  (your app)  │    │ (Anthropic,  │
│                 │    │              │    │  OpenAI, …)  │
└─────────────────┘    └──────────────┘    └──────────────┘
        ▲                      │
        │     validated PageIR │
        └──────────────────────┘
```

**Studio has no network access of its own.** It calls the callback
you supply. That callback is the only place credentials live, so the
host's backend stays in control of:

- Which provider is called.
- Which model and parameters the provider receives.
- Which users may call at all (auth, rate limits).
- Whether the response body ever reaches the client unfiltered.

Because the callback is a **function reference**, not a config
string, there is no way to embed an API key in a plugin option and
accidentally serialise it. If you return a value from a server
action, the secrets stay on the server.

<Aside type="caution" title="Secrets never leave the host backend">
	The full reasoning lives in
	[`docs/security/plugin-trust-model.md`](https://github.com/ancyloce/anvilkit-studio/blob/main/docs/security/plugin-trust-model.md)
	§3. This guide assumes you're reading from there for the formal
	capability boundaries.
</Aside>

## 2. The `generatePage` contract

```ts
import type {
	AiGenerationContext,
	PageIR,
} from "@anvilkit/core/types";

type GeneratePageFn = (
	prompt: string,
	ctx: AiGenerationContext,
) => Promise<PageIR>;
```

Input:

- `prompt` — the raw user string. No sanitisation is applied on the
  way in. Length-cap it yourself if your provider charges per input
  token.
- `ctx.availableComponents: AiComponentSchema[]` — an AI-safe
  projection of the host's Puck `Config`, built by
  `@anvilkit/schema/configToAiContext()`. This is the allow-list the
  LLM gets to pick from.
- `ctx.currentData?` — set only when you pass
  `forwardCurrentData: true` to the plugin; lets the LLM reference
  blocks already on the page.

Output: **a raw `PageIR` candidate**. It does not have to be valid
— the plugin passes it through `validateAiOutput()` before dispatch
(§4 below). A clean failure is preferable to a coerced success.

```ts
// minimum-viable generatePage (for tests or offline demos)
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

export const noopGeneratePage: GeneratePageFn = async (prompt, _ctx) => ({
	version: "1",
	root: {
		id: "root",
		type: "__root__",
		props: {},
		children: [
			{
				id: "hero-1",
				type: "Hero",
				props: { headline: prompt.slice(0, 80) },
			},
		],
	},
	assets: [],
	metadata: {},
});
```

## 3. Schema context

`@anvilkit/schema`'s `configToAiContext(puckConfig)` walks the same
Puck `Config` object you pass to `<Studio>` and produces a
deterministic, JSON-safe summary the LLM can reason about without
touching the live runtime:

```ts
import { configToAiContext } from "@anvilkit/schema";
import type { AiComponentSchema } from "@anvilkit/core/types";

const ctx = configToAiContext(puckConfig);
// ctx.availableComponents: AiComponentSchema[]
//   [
//     {
//       componentName: "Hero",
//       description: "Marketing hero block with headline + body.",
//       fields: [{ name: "title", description: "…", type: "text" }]
//     },
//     { componentName: "Pricing", description: "…", fields: [...] },
//     ...
//   ]
```

Two guarantees matter:

1. **The list is a closed set.** The LLM cannot conjure a component
   type that isn't in `availableComponents` — the validator will
   reject it (§4).
2. **The field kinds are AI-safe.** Non-serialisable fields (slots,
   renderers) are dropped by `extractFieldSchema`; the LLM sees only
   fields that can be round-tripped as JSON.

You normally do not pass `availableComponents` into the prompt
verbatim — shape it for your chosen provider's tool/function-calling
conventions (`@anvilkit/plugin-ai-copilot` gives you the data; how
you present it is provider-specific).

## 4. Validator boundary

Every `generatePage` response crosses one gate before the plugin
calls `puckApi.dispatch`:

```ts
import { validateAiOutput } from "@anvilkit/validator";

const validation = validateAiOutput(response, ctx.availableComponents);
if (!validation.valid) {
	// emit error, do NOT dispatch
}
```

The validator is a Zod schema. It enforces:

- `PageIR` shape (`version === "1"`, root node, assets array).
- Every child's `type` is in `availableComponents`.
- `props` is an object (serialisability constraints are documented
  in the trust model; see §4 there for the known gaps).

It does **not** enforce:

- The content of string props (sanitising is the exporter's job).
- Size limits (enforce in your `generatePage` if you care — see §6).
- Semantic sensibleness (an empty `headline` is syntactically valid).

A failure emits `ai-copilot:error` on the event bus with
`{ code: "VALIDATION_FAILED", issues }` and writes a structured log
line. The IR is dropped. No partial state is ever committed.

## 5. Error surfacing

The plugin emits three stable error codes:

| Code                | When                                                   |
| ------------------- | ------------------------------------------------------ |
| `VALIDATION_FAILED` | The response failed Zod validation.                    |
| `TIMEOUT`           | `generatePage` didn't resolve within `timeoutMs`.      |
| `GENERATE_FAILED`   | `generatePage` threw (network error, quota, etc).      |

Subscribe via any `StudioPluginContext` you hold:

```ts
import type { StudioPluginContext } from "@anvilkit/core/types";
import type { AiCopilotErrorPayload } from "@anvilkit/plugin-ai-copilot";

function wireErrorToast(ctx: StudioPluginContext, toast: (msg: string) => void) {
	ctx.on("ai-copilot:error", (payload: AiCopilotErrorPayload) => {
		if (payload.code === "TIMEOUT") {
			toast("AI generation timed out. Try again.");
		} else if (payload.code === "VALIDATION_FAILED") {
			toast(`Response rejected: ${payload.issues?.[0]?.message ?? "bad IR"}`);
		} else {
			toast(`Generation failed: ${payload.message}`);
		}
	});
}
```

Do not parse `payload.message` strings for logic — they are intended
for human display. The `code` field is the stable contract.

## 6. Cost controls

Four levers, in descending order of what the plugin provides vs.
what you must do yourself:

1. **`timeoutMs`** (plugin option, default 30 000 ms). After this,
   the outstanding generation is abandoned and a `TIMEOUT` error
   fires. This bounds pathological provider latency.
2. **`maxPromptLength`** (host-side, implement in your adapter).
   The plugin does not cap prompt length — runaway prompts bill
   against your quota before the LLM ever responds. A simple
   `if (prompt.length > 4_000) throw` in your `generatePage` is
   enough.
3. **Rate limiting** (host-side). Because `generatePage` is called
   from a host-owned function, you can rate-limit by user, by IP,
   by org — whatever auth context your backend has. Nothing in
   Studio can do this for you.
4. **Provider-side quotas.** The API key you hand the provider is
   the last line of defence. Use a scoped key with a spend cap.

## 7. Three worked adapters

Each adapter lives under `examples/ai-adapters/<provider>/` as a
standalone package with its own README. This guide shows the
interesting part — the `generatePage` body — inline, so drift is
caught by the CI harness.

### Anthropic (Claude)

```ts
import Anthropic from "@anthropic-ai/sdk";
import { validateAiOutput } from "@anvilkit/validator";
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

const client = new Anthropic({ apiKey: process.env.AI_API_KEY });

export const generatePage: GeneratePageFn = async (prompt, ctx) => {
	if (prompt.length > 4_000) {
		throw new Error("Prompt too long");
	}
	const response = await client.messages.create({
		model: "claude-sonnet-4-6",
		max_tokens: 4_096,
		system:
			"You are a page generator. Return ONLY a JSON PageIR. " +
			"Allowed component types: " +
			ctx.availableComponents.map((c) => c.componentName).join(", "),
		messages: [{ role: "user", content: prompt }],
	});
	const block = response.content[0];
	if (block?.type !== "text") {
		throw new Error("No text block in Claude response");
	}
	return JSON.parse(block.text);
};
```

Secrets: `process.env.AI_API_KEY` lives on the host backend. The
`generatePage` function is exported from a server module and
imported into the host's Studio config only in server components.

### OpenAI

```ts
import OpenAI from "openai";
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

const client = new OpenAI({ apiKey: process.env.AI_API_KEY });

export const generatePage: GeneratePageFn = async (prompt, ctx) => {
	if (prompt.length > 4_000) {
		throw new Error("Prompt too long");
	}
	const response = await client.chat.completions.create({
		model: "gpt-5",
		response_format: { type: "json_object" },
		messages: [
			{
				role: "system",
				content:
					"Return JSON PageIR only. Components: " +
					ctx.availableComponents.map((c) => c.componentName).join(", "),
			},
			{ role: "user", content: prompt },
		],
	});
	const text = response.choices[0]?.message?.content;
	if (!text) throw new Error("Empty OpenAI response");
	return JSON.parse(text);
};
```

### Generic `fetch` (provider-agnostic)

Use this shape if you front your provider through your own backend
route (recommended for browser-originating requests):

```ts
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

export function createFetchGeneratePage(endpoint: string): GeneratePageFn {
	return async (prompt, ctx) => {
		if (prompt.length > 4_000) {
			throw new Error("Prompt too long");
		}
		const response = await fetch(endpoint, {
			method: "POST",
			headers: { "content-type": "application/json" },
			body: JSON.stringify({
				prompt,
				availableComponents: ctx.availableComponents.map(
					(c) => c.componentName,
				),
			}),
		});
		if (!response.ok) {
			throw new Error(`AI endpoint returned ${response.status}`);
		}
		return (await response.json()) as Awaited<ReturnType<GeneratePageFn>>;
	};
}
```

The backend `POST /api/ai/generate-page` route then holds the real
API key and can enforce auth, rate limits, and prompt size in one
place.

## 8. Do not do this

These anti-patterns defeat the trust model — never ship a build
that contains any of them:

<Aside type="danger" title="Anti-patterns">
	**1. Hardcoding an API key client-side.**

	```ts
	// ❌ NEVER
	const client = new Anthropic({
		apiKey: "sk-ant-xxxxxxxxxxxxxxxxxxxx",
	});
	```

	Anyone who views page source sees the key. Rotate immediately.

	**2. Reading a secret from `import.meta.env.ANTHROPIC_API_KEY`
	inside code that runs in the browser.**

	Bundlers inline `VITE_*`, `NEXT_PUBLIC_*`, and similar prefixes
	into client bundles. If the env var ships to the browser, it is
	a client secret — treat it as leaked.

	**3. Embedding credentials in `studioConfig`.**

	`studioConfig` is visible to every registered plugin (trust
	model §2). Anything you put there is readable by a third-party
	plugin the host installs later. Keep secrets in module scope of
	your server-only `generatePage` module.

	**4. Trusting `prompt` as a server-to-LLM passthrough.**

	Prompts are user input. A prompt injection attack may try to
	coerce the LLM into returning JS-shaped data or component types
	the host did not declare. The validator rejects shape drift,
	but the adapter should still length-cap prompts and reject
	obvious abuse (see `validateAiOutput` §4).

	**5. Catching and swallowing `generatePage` errors silently.**

	Let them throw. The plugin turns thrown errors into a
	`GENERATE_FAILED` event so the UI can surface them. Swallowing
	the error makes "I clicked generate, nothing happened" the
	failure mode, which is much worse than a visible red message.
</Aside>
