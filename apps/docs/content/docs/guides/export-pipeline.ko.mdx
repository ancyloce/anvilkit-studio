AnvilKit은 모든 내보내기 — 오늘의 HTML, 내일의 React 소스와 PDF, 다음 주의 당신의 커스텀 형식 — 를 동일한 정규화된 `PageIR` 문서의 함수로 취급합니다. 이 가이드는 파이프라인을 처음부터 끝까지 살펴보고, 실제로 테스트된 JSON 내보내기를 작동 예제로 함께 제공합니다.

<Callout type="info" title="코드는 항상 그린을 유지한다">
	아래의 모든 코드 블록은
	[`apps/docs/tests/guides/export-pipeline.test.ts`](https://github.com/ancyloce/anvilkit-studio/blob/main/apps/docs/tests/guides/export-pipeline.test.ts)
	로 추출되어 CI에서 Vitest 테스트로 실행됩니다. `@anvilkit/core`,
	`@anvilkit/ir`, `@anvilkit/plugin-export-html`, 또는 작동 예제에서의
	드리프트는 문서가 배포되기 전에 빌드를 깨뜨립니다.
</Callout>

## 1. 파이프라인 개요

파이프라인에는 세 개의 명명된 단계와 각 쌍 사이의 한 번의 동결된 인계가 있습니다.

```text
Puck Data ──puckDataToIR()──▶ PageIR ──ExportFormatDefinition.run()──▶ { content, filename }
          (from @anvilkit/ir)  (from @anvilkit/core/types)       (string | Uint8Array)
```

- **1단계 — `Puck Data`**: Puck의 에디터가 당신에게 건네주는 것. 오서링 형태: 컴포넌트 id, UI 상태, 드래그 앤 드롭 메타데이터. 내보내기는 이것을 직접 보지 않습니다.
- **2단계 — `PageIR`**: 모든 내보내기가 소비하는, 정규화되고, 깊게 동결되고, JSON 안전한 트리. 내보내기마다 정확히 한 번
	[`puckDataToIR()`](/ko/api/ir/functions/puckdatatoir/) 에 의해 생성됩니다 — 함수
	속성은 경고로 폐기되고, 객체 키는 정규적으로 정렬되며,
	에셋은 루트 레벨 매니페스트로 수집됩니다.
- **3단계 — `ExportFormatDefinition.run(ir, options)`**: 당신의
	내보내기. 동결된 IR과 형식별 옵션을 받아,
	`string`(텍스트 형식) 또는
	`Uint8Array`(바이너리 형식)와 함께 권장 `filename`을 담은 `Promise<ExportResult>` 를 반환합니다.

이것이 작동하게 만드는 불변식은, IR 정규화 단계가 등록된 형식의 수에 관계없이 전체 파이프라인에서 **정확히 한 번** 실행된다는 것입니다. 동시에 등록된 두 내보내기는 바이트 단위로 동일한 입력을 봅니다. 이것이 바로 테스트 하니스가 하나의 정규화된 픽스처에서 두 개의 서로 다른 출력을 스냅샷할 수 있는 방식입니다.

## 2. `PageIR`의 형태

이 절의 모든 타입은 `@anvilkit/core/types`에서 가져온 `type` 내보내기입니다 — 런타임 코드 제로, React 제로, JS 레이어에서 `@puckeditor/core` 의존성 제로. `import type`로 가져오면 컴파일 타임에 지워집니다.

```ts
import type {
	ExportFormatDefinition,
	PageIR,
	PageIRAsset,
	PageIRMetadata,
	PageIRNode,
} from "@anvilkit/core/types";
```

### 2.1 루트 문서

```ts
import type { PageIR } from "@anvilkit/core/types";

const ir: PageIR = {
	version: "1",
	root: {
		id: "root",
		type: "__root__",
		props: {},
		children: [
			{
				id: "hero-1",
				type: "Hero",
				props: {
					headline: "Ship updates without friction.",
					description: "Deterministic exports for internal release pages.",
				},
			},
		],
	},
	assets: [],
	metadata: { createdAt: "2026-04-11T00:00:00.000Z" },
};
```

네 개의 최상위 키, 모두 필수:

| 키         | 형태                                 | 목적                                                                 |
| ---------- | ------------------------------------ | -------------------------------------------------------------------- |
| `version`  | 리터럴 `"1"`                         | 스키마 버전. 미래의 `"2"`는 IR 형태에 대한 호환성 깨짐 변경입니다.   |
| `root`     | [`PageIRNode`](/ko/api/core/interfaces/pageirnode/) | 합성된 `__root__` 노드; 그 `children`이 페이지 콘텐츠를 담습니다. |
| `assets`   | `readonly PageIRAsset[]`             | 트리에서 참조된 모든 에셋의 평탄화되고 중복 제거된 매니페스트.       |
| `metadata` | [`PageIRMetadata`](/ko/api/core/interfaces/pageirmetadata/) | 선택적 `title`/`description`/`createdAt`/`updatedAt`. |

### 2.2 노드

```ts
import type { PageIRNode } from "@anvilkit/core/types";

const heroNode: PageIRNode = {
	id: "hero-1",
	type: "Hero",
	props: {
		headline: "Ship updates without friction.",
		description: "Deterministic HTML exports.",
	},
	// `children` is omitted on leaf nodes (not set to []).
};
```

- `id` — 노드별 안정적인 식별자. Puck의 콘텐츠 id에서 이어받기 때문에 라운드트립이 무손실입니다.
- `type` — 노드가 생성된 출처인 컴포넌트 이름(`"Hero"`, `"Button"`, `"Section"`). 내보내기는 이를 기준으로 디스패치합니다.
- `props` — 불투명한 `Readonly<Record<string, unknown>>`. IR은 속성 형태에 대해 어떤 주장도 하지 않습니다; 그것은 내보내기의 일입니다.
- `children` — 선택적 재귀 배열. 리프에서는 생략됩니다(비어 있는 배열이 아님).
- `assets` — 선택적 노드 범위 에셋 목록. 전체 트리를 순회하지 않고 자기 몫만 해석하려는 내보내기를 위한 것.

### 2.3 에셋 참조

```ts
import type { PageIRAsset } from "@anvilkit/core/types";

const cover: PageIRAsset = {
	id: "a1b2c3",
	kind: "image",
	url: "https://cdn.example.com/cover.jpg",
	meta: { width: 1600, height: 900 },
};
```

`kind`는 **닫힌 유니온**입니다: `"image" | "video" | "font" | "script" |
"style" | "other"`. 설계상 이 유니온을 확장하는 것은 호환성 깨짐 변경입니다 — 타입 레벨에서 모든 경우를 처리하는 내보내기는 새로운 kind가 등장하면 재검토를 강제받습니다.

어떤 속성 키가 에셋이 되는가?
[`collectAssets()`](/ko/api/ir/functions/collectassets/) 는
`src`, `imageUrl`, `imageSrc`, `url`, `videoUrl`, `videoSrc`,
`fontUrl`, `scriptUrl`, `styleUrl`을 인식합니다. `href`는 의도적으로
제외됩니다 — 그것은 에셋이 아니라 내비게이션 링크입니다.

### 2.4 슬롯

Puck의 `slot` 타입 필드는 부모 노드의 `props` 백에 존재하는 자식 노드들의 중첩 배열로 IR에 나타납니다. 레이아웃 컴포넌트(예: `layout` 템플릿의 `leftColumn`/`rightColumn`)는 이에 의존합니다 — 빈 슬롯은 누락된 키가 아니라 `[]`입니다.

내보내기 시점에 어떤 `type`의 어떤 속성 키가 슬롯인지 알아야 할 때
[`identifySlots(config)`](/ko/api/ir/functions/identifyslots/) 를 사용하세요. IR 패키지는 정규화 중에 슬롯 배열과 일반 데이터 배열을 구분하기 위해 이를 사용합니다.

## 3. 새 내보내기 작성하기

내보내기는 세 가지로 구성됩니다:

1. `ExportFormatDefinition` 객체 — 형식별 디스크립터.
2. `register()` 반환값의 `exportFormats` 배열을 통해 정의를 기여하는 `StudioPlugin`.
3. (선택적으로) Studio 툴바에서 내보내기를 구동하는 `StudioHeaderAction`.

이 가이드의 작동 예제는
[`@anvilkit/example-export-json`](https://github.com/ancyloce/anvilkit-studio/tree/main/examples/plugins/export-json)
— 최소한의 JSON 내보내기입니다. 그 전체 `run()`은 한 화면에 들어맞습니다:

```ts
import type {
	ExportFormatDefinition,
	PageIR,
} from "@anvilkit/core/types";

interface JsonExportOptions extends Record<string, unknown> {
	readonly indent?: number;
	readonly filename?: string;
}

export const jsonFormat: ExportFormatDefinition<JsonExportOptions> = {
	id: "json",
	label: "JSON",
	extension: "json",
	mimeType: "application/json",
	async run(ir: PageIR, options) {
		const indent = options.indent ?? 2;
		const filename = options.filename ?? "page.json";
		const content = JSON.stringify(ir, null, indent);
		return { content, filename };
	},
};
```

이를 `StudioPlugin`으로 감쌉니다:

```ts
import type { StudioPlugin, StudioPluginMeta } from "@anvilkit/core/types";
import { jsonFormat } from "./json-format.js";

const meta: StudioPluginMeta = {
	id: "anvilkit-example-export-json",
	name: "JSON Export",
	version: "0.1.0",
	coreVersion: "^0.1.0-alpha",
};

export function createExportJsonPlugin(): StudioPlugin {
	return {
		meta,
		register(_ctx) {
			return {
				meta,
				exportFormats: [jsonFormat],
			};
		},
	};
}
```

그런 다음 런타임에 등록합니다:

```ts
import { createStudioConfig } from "@anvilkit/core";
import { createExportJsonPlugin } from "@anvilkit/example-export-json";

const studioConfig = createStudioConfig({
	plugins: [createExportJsonPlugin()],
});
```

호스트 앱은 id를 통해
`useExportStore.exportAs("json", { indent: 0 })` 로 그 형식을 트리거합니다. 런타임은
`runtime.exportFormats`에서 그 id를 찾아,
갓 정규화된 IR로 `run(ir, options)`를 호출하고, 결과를
다운로드용 `Blob`으로 돌려줍니다.

<Callout type="warn" title="`run`은 Data가 아니라 IR을 받는다">
	`ExportFormatDefinition.run`은
	[`PageIR`](/ko/api/core/interfaces/pageir/) 에 대해 타입이 지정되어 있습니다 — Puck `Data`를
	받는 `run`을 물리적으로 타입 검사할 수 없습니다. 그것이 바로
	IR 레이어가 존재하는 유일한 이유입니다: 내보내기는 실수로
	정규화를 건너뛸 수 없습니다.
</Callout>

## 4. HTML 내보내기 둘러보기

[`@anvilkit/plugin-export-html`](https://github.com/ancyloce/anvilkit-studio/tree/main/packages/plugins/plugin-export-html)
은 레퍼런스 내보내기입니다. 당신만의 것을 만들기 시작할 때, 그 `format-definition.ts`가 가장 먼저 읽어야 할 파일입니다:

```ts
// packages/plugins/plugin-export-html/src/format-definition.ts
export const htmlFormat: ExportFormatDefinition<HtmlExportOptions> = {
	id: "html",
	label: "HTML",
	extension: "html",
	mimeType: "text/html",
	run: async (ir, options) => {
		const ctx = makeEmitContext();
		const { html, usedClassnames, warnings: htmlWarnings, emittedAssetIds } = emitHtml(ir, options, ctx);
		const { inlined, warnings: assetWarnings } = await inlineAssets(ir.assets, {
			thresholdBytes: options.inlineAssetThresholdBytes ?? 32_768,
			...(options.fetchAsset ? { fetchAsset: options.fetchAsset } : {}),
			emittedAssetIds,
		});
		const css = emitCss(usedClassnames, options);
		const bodyHtml = substituteAssets(html, inlined);
		const content = wrapDocument({
			title: options.title ?? ir.metadata.title ?? "Exported Page",
			description: ir.metadata.description,
			css,
			bodyHtml,
		});
		return { content, filename: "page.html", warnings: [...htmlWarnings, ...assetWarnings] };
	},
};
```

다섯 가지 책임, 각각 자체 모듈에:

| 모듈                     | 책임                                                               |
| ------------------------ | ------------------------------------------------------------------ |
| `emit-html.ts`           | IR을 순회하고 노드 `type`별로 클래스 장식된 HTML 문자열을 방출한다. |
| `emit-css.ts`            | `emit-html`이 사용한 클래스명에 대한 CSS 규칙을 방출한다.          |
| `inline-assets.ts`       | 선택적으로 에셋 바이트를 가져오고 임계값 이하의 방출된 이미지를 base64 인코딩한다. |
| `substitute-assets.ts`   | body의 `asset-id://…` 자리표시자를 data URI로 교체한다.            |
| `wrap-document.ts`       | body + CSS를 독립적인 `<!doctype html>` 문서로 감싼다.            |

전체 파이프라인은 의도적으로 순수합니다 — `run()`이 `async`인 이유는
에셋 해석과 호스트가 제공하는 인라인화가 외부
콜백을 await할 수 있기 때문입니다. 그 외의 모든 것은 깊게 동결된 IR에 대한 동기적 문자열 조작입니다.

플러그인 셸은 얇은 래퍼입니다:

```ts
// create-html-export-plugin.ts — abbreviated
const htmlExportPluginMeta: StudioPluginMeta = {
	id: "anvilkit-plugin-export-html",
	name: "HTML Export",
	version: "0.1.0-alpha.0",
	coreVersion: "^0.1.0-alpha",
};

export function createHtmlExportPlugin(opts: HtmlExportOptions = {}): StudioPlugin {
	return {
		meta: htmlExportPluginMeta,
		register(_ctx) {
			return {
				meta: htmlExportPluginMeta,
				exportFormats: [htmlFormat],
				...(opts.buildIR ? { headerActions: [createExportHtmlHeaderAction(htmlFormat, opts)] } : {}),
			};
		},
	};
}
```

당신이 기여하는 모든 것 — `exportFormats`, `headerActions`, 그리고
(표시되지 않은) `hooks` / `overrides` — 은 평범한 데이터입니다. 런타임이
마운트 시점에 그것을 조합합니다; 당신의 플러그인 코드는 결코
React 트리에 손을 뻗지 않습니다.

## 5. 에셋 인라인화 전략

텍스트 중심 내보내기(HTML, MDX, React 소스)는 보통 출력을 단일한 자기 완결형 파일로 만들기 위해 어느 정도의 인라인화를 원합니다. 바이너리
내보내기(PDF, ZIP)는 항상 인라인화합니다. 순수 데이터 내보내기(JSON)는 결코
인라인화하지 않습니다. 세 가지 전략은 바이트가 최종적으로 어디에 놓이는지에서 차이가 납니다.

### 5.1 Base64 인라인화 (호스트가 제어하는 HTML)

`@anvilkit/plugin-export-html`은 디코딩된 크기가
`inlineAssetThresholdBytes`(기본값 `32_768`) 미만인 방출된 이미지를 무엇이든 인라인화할 수 있지만,
호스트가 `fetchAsset`을 제공할 때만 바이트를 가져옵니다. `inlineAssets`
헬퍼는 형태를 단순하게 유지합니다 — `Map<assetId, dataUrl>`
— 그리고 HTML 파이프라인은 출력하는 길에 교체합니다:

```ts
// Simplified from inline-assets.ts.
async function inlineSmallImages(assets, { thresholdBytes, fetchAsset, emittedAssetIds }) {
	const inlined = new Map<string, string>();
	if (!fetchAsset) return inlined;
	for (const asset of assets) {
		if (!emittedAssetIds.has(asset.id)) continue;
		if (asset.kind !== "image") continue;
		const { bytes, contentType } = await fetchAsset(asset.url, { maxBytes: thresholdBytes });
		if (bytes.length > thresholdBytes) continue;
		inlined.set(asset.id, `data:${contentType};base64,${toBase64(bytes)}`);
	}
	return inlined;
}
```

다음의 경우에 이것을 사용하세요:

- 호스트가 단일 파일을 다운로드하고 에셋을 다시 가져올 수 없는 경우.
- 에셋이 작은 경우(아이콘, favicon, 32 KB 미만 이미지).
- 파일이 오프라인에서 작동하기를 원하는 경우.

### 5.2 URL로 참조

에셋 URL을 그대로 두고 평범한 `<img src="…">`(또는
이에 상응하는 것)를 방출합니다. 다음의 경우에 가장 좋습니다:

- 에셋이 큰 경우(hero 이미지, 동영상).
- 내보내기가 원래 호스트에 도달할 수 있는 환경에서 소비되는 경우.
- 이식성보다 출력 크기를 더 신경 쓰는 경우.

기본 HTML 파이프라인은 URL을 참조합니다. 호스트가
페처를 제공하면, 파이프라인은 둘을 혼합합니다: 임계값 이하는 인라인화하고,
나머지는 참조합니다. `MISSING_INLINE` 경고(레벨 `"warn"`)는 임계값에
미치지 못한 것을 짚어내어, 호스트가
예산을 조이거나 에셋을 다시 호스팅할지 결정할 수 있게 합니다.

### 5.3 참조를 완전히 제거

일부 내보내기(다운스트림 파이프라인으로 전달되는 JSON 스냅샷, 별도의 에셋 패스로 재구성되는 MDX
번들)는 IR의
`assets` 매니페스트를 그대로 복사해 가기만을 원합니다. 이
가이드의 JSON 내보내기가 정전적인 예입니다 — 그것은 가져오지도 인라인화하지도 않고,
`ir.assets`를 노드 트리와 나란히 직렬화하기만 합니다.

### 5.4 에셋 경고 분류 체계

모든 내보내기는 치명적이지 않은 발견에 대해 예외를 던지는 대신
`ExportWarning[]`을 방출해야 합니다. HTML 플러그인은 다음과 같은 경고 코드를 사용합니다:

- `MISSING_INLINE` — 에셋이 임계값을 초과하여 URL로 남겨졌다.
- `ASSET_FETCH_FAILED` — `fetchAsset()`이 던졌다; URL은 그대로 남겨졌다.
- `ASSET_FETCH_BLOCKED` — 기본 페처가 http(s)가 아닌 URL을 거부했다.
- `UNINLINEABLE_ASSET_KIND` — 이미지가 아닌 에셋은 data-URI 인라인화 대상이 아니었다.
- `ASSET_UNRESOLVED` — 등록된 리졸버 중 어느 것도 `asset://` URL을 소유하지 않았다.
- `UNSUPPORTED_CHILDREN` — 렌더링된 컴포넌트에 내보내기가 아직 보존할 수 없는 중첩된 IR 자식이 있었다.

하드 실패(잘못된 IR, 수정 불가능한 인코딩 오류)는
`StudioExportError`를 던져야 하며, 런타임이 이를 잡아 호스트에 노출합니다.
사용자가 복구할 수 없는 것에 대해 `level: "error"` 경고를 결코 반환하지 마세요.

## 6. 라운드트립 불변식

파이프라인에서 가장 중요한 단일 불변식은 다음과 같습니다:

```text
irToPuckData(puckDataToIR(d)) ≡ d   // up to root-prop canonicalization
```

[`irToPuckData`](/ko/api/ir/functions/irtopuckdata/) 는
`puckDataToIR`의 역입니다. 둘은 함께 IR이 그것을 생성한
오서링 `Data`에 대해 무손실임을 증명합니다. 세 가지 구체적인 이점:

1. **AI가 생성한 페이지가 라운드트립한다.** AI-copilot 플러그인은
	그 LLM 출력을 `PageIR`로 검증한 다음,
	`irToPuckData`를 통해 그것을 `setData` 디스패치로 보냅니다. 이 불변식이 없으면,
	AI 편집은 생성할 때마다 조금씩 드리프트할 것입니다.
2. **스냅샷 테스트가 바이트 단위로 안정적으로 유지된다.** 동일한 `Data`에 대해 `puckDataToIR`을 두 번 실행하면
	바이트 단위로 동일한 JSON이 생성됩니다(속성은
	정규적으로 정렬되고, 타임스탬프는 주입 가능). 그것이 바로
	HTML 플러그인의 골든 스냅샷이 CI 실행 전반에 걸쳐 `toMatchFileSnapshot()`을
	통과하게 만드는 것입니다.
3. **두 내보내기는 "페이지에 무엇이 있는가"에 대해 의견이 어긋날 수 없다.** 둘 다
	같은 IR을 소비하므로, Puck의 특이점이나
	오서링 전용 필드에서 갈라질 수 없습니다.

불변식을 명시적으로 테스트하세요 — 모든 새 내보내기는 대표적인 픽스처에 대해 그것을 증명하는
`round-trip.test.ts`를 함께 제공해야 합니다:

```ts
import { irToPuckData, puckDataToIR } from "@anvilkit/ir";
import { expect, it } from "vitest";

it("round-trips the demo data through IR without loss", () => {
	const FIXED_CLOCK = () => new Date("2026-04-11T00:00:00.000Z");
	const ir = puckDataToIR(demoData, demoConfig, { now: FIXED_CLOCK });
	expect(irToPuckData(ir)).toEqual(demoData);
});
```

`FIXED_CLOCK`이 핵심입니다 — `metadata.createdAt`은 ISO 문자열이며,
유동적인 `new Date()`는 스냅샷 안정성을 무너뜨릴 것입니다. 파이프라인을
실행하는 모든 AnvilKit 테스트는 `puckDataToIR`의 `opts.now`를 통해
주입된 시계를 통과시킵니다.

## 7. 내보내기 테스트하기

`@anvilkit/plugin-export-html`은 세 개의 중첩되는 테스트 레이어를 사용합니다.
그것들을 복사하세요 — 그것들은 세 가지 다른 종류의 회귀를 잡아냅니다.

### 7.1 골든 스냅샷

픽스처당 하나의 테스트, 각각은 내보내기 출력이 커밋된
`.snap.html` 파일과 일치함을 단언합니다:

```ts
// Adapted from plugin-export-html/src/__tests__/golden.test.ts
import { fileURLToPath } from "node:url";
import { expect, it } from "vitest";
import { htmlFormat } from "../format-definition.js";
import { heroFixture } from "./__fixtures__/hero.fixture.js";

const fetchAsset = async () => ({
	bytes: new Uint8Array([1, 2, 3, 4]),
	contentType: "image/jpeg",
});

it("renders the hero fixture to a stable html document", async () => {
	const result = await htmlFormat.run(heroFixture, {
		inlineStyles: true,
		inlineAssetThresholdBytes: 0,
		title: "Hero",
		fetchAsset,
	});
	expect(result.content).toMatchFileSnapshot(
		fileURLToPath(new URL("./__snapshots__/hero.snap.html", import.meta.url)),
	);
});
```

`toMatchFileSnapshot`은 첫 실행 시 작성하고 이후
실행 시 비교합니다. 내보내기, IR 형태, 또는 픽스처 기본값에서의 드리프트는 모두
`pnpm test -u`를 통해 명시적으로 수락해야 하는 차이를 생성합니다.

### 7.2 라운드트립 파싱

골든 스냅샷은 출력의 *형태*를 고정하지만,
"HTML은 기술적으로는 유효하나 headline을 잃어버렸다"를 잡아낼 수는 없습니다. 출력을
`happy-dom`으로 파싱하고 다시 쿼리하세요:

```ts
import { Window } from "happy-dom";
import { expect, it } from "vitest";

it("preserves hero headline content", async () => {
	const { content } = await htmlFormat.run(heroFixture, {
		inlineStyles: true,
		inlineAssetThresholdBytes: 0,
		fetchAsset: async () => ({
			bytes: new Uint8Array([1, 2, 3, 4]),
			contentType: "image/jpeg",
		}),
	});
	const window = new Window();
	window.document.write(content as string);
	window.document.close();
	expect(
		window.document.querySelector(".ak-hero__headline")?.textContent,
	).toContain("Ship updates without friction.");
});
```

### 7.3 등록 테스트

세 번째 레이어는 플러그인이 깔끔하게 연결됨을 단언합니다.
`compilePlugins([yourPlugin()], ctx)`를 호출하고, 당신의 형식 id가
`runtime.exportFormats`에 나타나는지 확인하세요:

```ts
import { StudioConfigSchema, compilePlugins } from "@anvilkit/core";
import type { StudioPluginContext } from "@anvilkit/core/types";
import { expect, it, vi } from "vitest";
import { createExportJsonPlugin } from "@anvilkit/example-export-json";

const studioConfig = StudioConfigSchema.parse({});

function makeCtx(): StudioPluginContext {
	return {
		getData: () => ({ root: { props: {} }, content: [], zones: {} }),
		getPuckApi: (() => {
			throw new Error("getPuckApi unused in this test");
		}) as unknown as StudioPluginContext["getPuckApi"],
		studioConfig,
		log: vi.fn(),
		emit: vi.fn(),
	};
}

it("registers the json export format via compilePlugins", async () => {
	const runtime = await compilePlugins([createExportJsonPlugin()], makeCtx());
	expect([...runtime.exportFormats.keys()]).toContain("json");
});
```

JSON 예제는 세 가지 모두를 함께 제공합니다 — 전체 스위트와 함께
`apps/demo/lib/puck-demo.ts`를 미러링하는 데모 페이지 스냅샷에 대해서는,
[`examples/plugins/export-json/src/__tests__/`](https://github.com/ancyloce/anvilkit-studio/tree/main/examples/plugins/export-json/src/__tests__)
를 참조하세요.

<Callout type="info" title="하나의 파이프라인, 세 개의 테스트, 하나의 픽스처">
	세 개의 테스트 레이어 모두를 같은 IR 픽스처로 구동되게 유지하세요.
	픽스처가 바뀌면, 단일 진실 공급원으로부터 세 개의 구별된 신호 — 출력
	형태(골든), 의미적 보존(라운드트립 파싱), 그리고
	플러그인 배선(등록) — 을 얻습니다.
</Callout>
