모든 `@anvilkit/<slug>` 컴포넌트 패키지는 동일한
제너레이터 `pnpm -C packages/components gen:component` 에서 시작합니다. 이는
아홉 개 파일 스캐폴드 — `package.json`, `rslib.config.ts`, `tsconfig.json`,
`src/index.ts`, `src/<ComponentName>.tsx`, `src/config.ts`,
`src/styles.css`, `src/styles.d.ts`, `README.md` — 를 작성하며,
워크스페이스의 Rslib, Tailwind, Puck 규약에 연결합니다. 새로운 기여자는
클론에서 이름이 지정되고 빌드 가능한 패키지까지 5분 이내에
도달할 수 있어야 합니다.

이 가이드는 단계별 안내입니다. 스캐폴딩된 코드가
지켜야 하는 계약에 대해서는 [컴포넌트 작성](/ko/guides/component-authoring/)을 참조하세요.

<Callout type="info" title="훑어보기 순서">
	이미 저장소를 클론했다면, 대화형 실행은 §2로,
	플래그 기반 실행은 §3으로 바로 건너뛰세요. §6은 완전히 새로운 `@anvilkit/callout`
	패키지를 위한 전부 재현 가능한 실전 예제입니다.
</Callout>

## 1. 사전 요구 사항

컴포넌트 워크스페이스는 자체 pnpm
워크스페이스를 가진 git 서브모듈입니다. 서브모듈 초기화 단계를 놓치면 제너레이터는
실행을 거부합니다.

1. **서브모듈과 함께 클론하기.**

	```bash
	git clone --recurse-submodules https://github.com/ancyloce/anvilkit-studio.git
	cd anvilkit-studio
	```

	이미 `--recurse-submodules` 없이 클론했다면, 저장소 루트에서 한 번
	`git submodule update --init --recursive` 를 실행하세요.

2. **툴체인 고정하기.** 이 저장소는
	`packageManager` 를 통해 pnpm 10.33.0 을 강제하고, Node ≥ 18 을 대상으로 합니다.

	```bash
	corepack enable
	node -v        # ≥ 18
	pnpm -v        # 10.33.0
	```

3. **의존성 설치하기.** 이를 저장소 루트에서 실행하세요. Turbo 와
	pnpm 워크스페이스가 `packages/components/` 를 자동으로 인식합니다.

	```bash
	pnpm install
	```

이 시점에서 `packages/components/src/` 에는 기존
컴포넌트 패키지(`button`, `hero`, `navbar` 등)가 들어 있고, 제너레이터는
하나를 더 추가할 준비가 되어 있습니다.

## 2. 대화형 실행

기본 호출은 네 개의 답변을 묻고, 각각을 검증한 뒤
스캐폴드를 작성합니다. `-C` 플래그가 명령을
컴포넌트 워크스페이스로 한정하도록 저장소 루트에서 실행하세요.

```bash
pnpm -C packages/components gen:component
```

프롬프트는 순서대로:

| 프롬프트                | 허용 값                                                     | 기본값                          |
| ----------------------- | ----------------------------------------------------------- | ------------------------------- |
| **컴포넌트 slug**       | 소문자 kebab-case, `src/` 아래에서 고유                      | _필수_                          |
| **표시 레이블**         | Puck 레일에 표시되는 자유 형식 문자열                        | `Title-Cased Slug`(slug 에서)   |
| **스캐폴드 유형**       | `content` · `layout` · `form`                               | _필수_                          |
| **제안 카테고리**       | 선택적 소문자 slug(`marketing`, `forms` 등)                | 비어 있음                       |

검증 규칙(`turbo/generators/config.ts` 에 의해 강제됨):

- slug 는 `^[a-z0-9]+(?:-[a-z0-9]+)*$` 에 일치해야 합니다 — 즉 `button-group` 은
	괜찮지만 `ButtonGroup` 과 `my_button` 은 안 됩니다.
- slug 는 고유해야 합니다 — `src/<slug>/` 가 이미 존재하면 제너레이터는
	그것을 덮어쓰기를 거부합니다.
- 카테고리는(제공된 경우) 동일한 kebab-case 형식을 만족해야 합니다.

성공적인 실행은 파일마다 한 줄의 `>>> Added "…"` 를 출력하고 깔끔하게 종료합니다.

## 3. 플래그 기반 실행

스크립트화 가능한 호출은 프롬프트를 건너뜁니다. pnpm 이 플래그를
스스로 소비하는 대신 제너레이터로 전달하도록 `--` 구분자가
필요합니다.

```bash
pnpm -C packages/components gen:component -- \
	--name button-group \
	--label "Button Group" \
	--template content \
	--category actions
```

지원되는 플래그:

| 플래그        | 필수     | 비고                                                        |
| ------------- | -------- | ----------------------------------------------------------- |
| `--name`      | 예       | 컴포넌트 slug. 대화형 프롬프트와 동일한 검증.               |
| `--label`     | 아니요   | 기본값은 Title Case 의 slug.                                |
| `--template`  | 아니요   | 기본값은 `content`. `content`, `layout`, `form` 중 하나.    |
| `--category`  | 아니요   | `src/config.ts` 의 `metadata.suggestedCategory` 에 기록됨.  |

잘못된 입력은 검증기의 메시지와 함께 즉시 실패합니다 — 스캐폴드는
작성되지 않으며, 부분 상태도 남지 않습니다.

## 4. 세 가지 템플릿

모든 템플릿은 동일한 아홉 개 파일을 생성하지만,
`<ComponentName>.tsx`, `config.ts`, `package.json` 의 형태가 달라
스캐폴드가 컴포넌트의 의도에 맞도록 합니다. 당신이
만들려는 것에 가장 잘 맞는 것을 고르세요.

<Tabs items={['content', 'layout', 'form']}>

<Tab value="content">

**언제 고를까.** 정적 마케팅 표면 — 버튼, 히어로,
섹션, 가격 블록. 이것이 기본값이며 대부분의 `@anvilkit/*` 패키지에
가장 가까운 일치입니다.

**무엇을 스캐폴딩하는가.**

- **속성:** `title: string`, `body: RichText`, `alignment?: 'left' | 'center' | 'right'`.
- **필드:** `text`(title), `richtext`(body), `radio`(alignment).
- **`package.json` peer deps:** `@puckeditor/core`, `react`, `react-dom`.
- **`@anvilkit/ui` 의존성 없음.**

**렌더링 형태**(`templates/component/content/src/component.tsx` 에서 그대로):

```tsx
import type { RichText } from "@puckeditor/core";

export interface CalloutProps {
	title: string;
	body: RichText;
	alignment?: "left" | "center" | "right";
}

export interface CalloutViewProps extends CalloutProps {
	editMode?: boolean;
}
```

</Tab>

<Tab value="layout">

**언제 고를까.** 자식 컴포넌트를 끼우는 구조적 컨테이너 —
벤토 그리드, 2단 섹션, 페이지 셸. 컴포넌트의
역할이 종단 콘텐츠 자체를 렌더링하는 것이 아니라
다른 Puck 컴포넌트를 배치하는 것일 때 이것을 사용하세요.

**무엇을 스캐폴딩하는가.**

- **속성:** `title: string`, `leftColumn: Slot`, `rightColumn: Slot`, `emphasis?: 'balanced' | 'accent'`.
- **필드:** `text`(title), `slot` × 2(좌/우 열), `radio`(emphasis).
- **렌더링:** 설정의 `render` 는 각 `Slot` 을
	`createElement(LeftColumn)` / `createElement(RightColumn)` 으로 감싸 Puck 이
	중첩된 드롭 존을 마운트하게 합니다.
- **`package.json` peer deps:** `content` 와 동일.

렌더링된 컴포넌트는 `leftColumn`/`rightColumn` 을
`ReactNode` 로 받고 `editMode` 를 관통시킵니다 — 스캐폴드는 이미
편집 중에만 표시되는 "Layout" 어포던스 레이블을 연결해 둡니다.

</Tab>

<Tab value="form">

**언제 고를까.** 에디터 인식 비활성 상태가 필요한
대화형 입력 — 텍스트 입력, 셀렉트, 폼 필드 래퍼. 이것은
`@anvilkit/ui` 에 런타임 의존성을 가지는 유일한 템플릿입니다.

**무엇을 스캐폴딩하는가.**

- **속성:** `label`, `name`, `type?`(text/email/password/search/tel/url), `placeholder?`, `helperText?`, `defaultValue?`, `required?`, `disabled?`.
- **필드:** `text`, `text`, `select`, `text`, `textarea`, `text`, `radio`, `radio`.
- **`package.json` peer deps:** 일반적인 Puck + React 와 함께 `@anvilkit/ui` 를 추가합니다.
- **`styles.css`:** Tailwind 가 상위 원시 요소의 클래스를 인식하도록
	추가 `@source '../../../../ui/src/input.tsx'` 를 포함합니다.
- **`editMode` 처리:** 뷰 컴포넌트는 `@anvilkit/ui` 의
	`Input` 을 `readOnly={editMode}` 와 차분한 배경으로 감싸,
	필드가 에디터 안에서 스스로 중립화되도록 합니다.

</Tab>

</Tabs>

### 공통 스캐폴드 파일

템플릿과 무관하게, 모든 스캐폴드에는 다음이 포함됩니다:

- **`src/config.ts`** — `metadata`, `defaultProps`, `fields`,
	`render<ComponentName>` 어댑터, `<slug>Config`, `componentConfig` 를 내보냅니다.
	`metadata.packageName` 은 `package.json` 에서 가져오므로 이름 변경에 안전하며,
	`metadata.componentSlug` 는 제너레이터의 답변으로부터 작성됩니다.
- **`src/index.ts`** — 컴포넌트, 그것의 `Props` +
	`ViewProps` 타입, 그리고 네 개의 정규 심볼(`componentConfig`,
	`defaultProps`, `fields`, `metadata`)을 원래 이름과
	lowerCamel 별칭 둘 다로 다시 내보냅니다.
- **`src/styles.css`** — `@import '@anvilkit/tailwind-config/shadcn'`
	와 로컬 `@source './**/*.{ts,tsx}'` 디렉티브로 시작합니다.
- **`package.json`** — 듀얼 CJS/ESM `exports`, `./styles.css`
	하위 경로, 그리고 Rslib 기반의 `build` / `dev` / `typecheck` 스크립트 세트.

## 5. 스캐폴딩 후 체크리스트

제너레이터는 동작하는 스텁 — lint, typecheck, 빌드를 통과 —
을 작성하지만, 그 스텁은 의도적으로 최소한입니다. 패키지를 마무리하는 일은
매번 동일한 5단계 루프를 따릅니다.

1. **렌더링 구현하기.** `src/<ComponentName>.tsx` 를 열고
	플레이스홀더 마크업을 실제 컴포넌트로 교체하세요. 직렬화 가능한
	속성과 `editMode` 의 단일 진실 공급원으로서
	`<ComponentName>ViewProps` 형태를 유지하세요.

2. **`defaultProps`, `fields`, `metadata` 확정하기.**
	`src/config.ts` 에서 기본값을 데모에서 의도적으로
	보이는 것으로 조정하고 필드 스키마를 다듬으세요 — enum 이 늘어나면
	`radio`→`select` 로 바꾸고, 반복 가능한 그룹에는 `array`/`object` 필드를
	추가하세요. 비어 있었다면 `metadata.suggestedCategory` 를 설정하세요.

3. **peer deps + `@source` 항목 연결하기.** `@anvilkit/ui`
	원시 요소를 사용하기 시작하면, 그것을 `peerDependencies` +
	`devDependencies` 에 추가하고 그 소스 경로를 추가 `@source` 디렉티브로
	`styles.css` 에 덧붙이세요.

4. **검증하기.** 워크스페이스에는 단일 스모크 명령이 있습니다:

	```bash
	pnpm -C packages/components lint \
		&& pnpm -C packages/components typecheck \
		&& pnpm -C packages/components build
	```

	CI 는 동일한 세 가지에 더해 `madge`(순환 의존성), `publint`
	(패키지 내보내기), Playwright 스위트를 실행합니다.

5. **데모에 통합하기.** `apps/studio/` 에서 손볼 두 파일:

	- **`apps/studio/lib/puck-demo.ts`** — import,
		`DemoComponents` 항목, 카테고리 연결,
		`demoConfig.components` 항목, 그리고 `createDemoData` 의 항목을 추가하세요.
	- **`apps/studio/next.config.js`** — Next 가 소스
		패키지를 트랜스파일할 수 있도록 `@anvilkit/<slug>` 를
		`transpilePackages` 배열에 추가하세요.

	`pnpm -C apps/studio dev` 를 실행하고 컴포넌트가
	`/puck/editor` 의 Puck 에디터에 드롭되는지 확인하세요.

## 6. 게시 워크플로

버전 관리는 Changesets 기반입니다. 각 `@anvilkit/<slug>` 는
독립적으로 릴리스됩니다.

1. **changeset 추가하기.** 컴포넌트 워크스페이스에서:

	```bash
	pnpm -C packages/components changeset
	```

	새 패키지를 선택하고, bump 수준(완전히 새로운 패키지에는 보통
	`minor`)을 선택한 뒤, 사용자 대상 설명을 작성하세요 — 그것이 그대로
	변경 로그 항목이 됩니다.

2. **PR 열기.** changeset 파일을 컴포넌트
	구현과 함께 커밋하세요. CI 는 푸시마다 `lint`, `typecheck`, `madge`, `test`,
	`build`, `publint`, Playwright E2E 스위트를 실행합니다 —
	[`.github/workflows/ci.yml`](https://github.com/ancyloce/anvilkit-studio/blob/main/.github/workflows/ci.yml) 를 참조하세요.

3. **`main` 에 머지하기.** Changesets 의 GitHub action 이 영향을 받는
	`package.json` 버전을 bump 하고 패키지별 `CHANGELOG.md` 항목을
	작성하는 release PR 을 엽니다(또는 업데이트합니다).

4. **release PR 머지하기.** 그 action 은 그 후 다음을 실행합니다:

	```bash
	pnpm -C packages/components release
	# = changeset version && pnpm build && changeset publish
	```

	보류 중인 changeset 이 있는 각 패키지가 새 버전으로
	npm 에 게시됩니다. 소비자는 필요한 것만 설치합니다 —
	`pnpm add @anvilkit/<slug> @puckeditor/core`.

<Callout type="warn" title="버전을 손으로 편집하지 말 것">
	패키지 버전은 `packages/components/src/<slug>/package.json` 에 있지만,
	그것들을 직접 편집하면 Changesets 의 CHANGELOG 생성을 우회하고
	release PR 을 비동기화합니다. bump 는 항상
	`pnpm changeset` 을 통해 처리하세요.
</Callout>

## 7. 실전 예제 — `@anvilkit/callout`

이는 완전히 새로운 `callout` 컴포넌트를 빈
`src/callout/` 디렉터리에서 데모 통합까지 출시하는 과정을 따라갑니다. 새로운
기여자는 15분 이내에 이를 처음부터 끝까지 재현할 수 있습니다.

### 7.1 스캐폴딩

저장소 루트에서, `pnpm install` 이후에:

```bash
pnpm -C packages/components gen:component -- \
	--name callout \
	--label "Callout" \
	--template content \
	--category marketing
```

실행 후, `packages/components/src/callout/` 에는
아홉 개 파일 스캐폴드가 들어 있고, 생성된 `src/config.ts` 에는
`metadata.suggestedCategory: 'marketing'` 이 이미 연결되어 있습니다.

### 7.2 렌더링 구현하기

`packages/components/src/callout/src/Callout.tsx` 를 톤 인식
컨테이너로 교체하세요. `tone` 속성은 shadcn 시맨틱 token
세트를 골라, 새 색상을 만들지 않고도 컴포넌트를
테마 인식으로 만듭니다:

```tsx
import type { RichText } from "@puckeditor/core";

export interface CalloutProps {
	title: string;
	body: RichText;
	tone?: "info" | "success" | "warning" | "danger";
}

export interface CalloutViewProps extends CalloutProps {
	editMode?: boolean;
}

const toneClasses: Record<
	NonNullable<CalloutProps["tone"]>,
	{ border: string; bg: string; text: string }
> = {
	info: {
		border: "border-border",
		bg: "bg-muted/40",
		text: "text-foreground",
	},
	success: {
		border: "border-primary/30",
		bg: "bg-primary/10",
		text: "text-primary",
	},
	warning: {
		border: "border-accent",
		bg: "bg-accent/20",
		text: "text-accent-foreground",
	},
	danger: {
		border: "border-destructive/40",
		bg: "bg-destructive/10",
		text: "text-destructive",
	},
};

export function Callout({
	title,
	body,
	tone = "info",
}: CalloutViewProps) {
	const classes = toneClasses[tone];

	return (
		<aside
			role="note"
			className={`grid gap-3 rounded-2xl border p-5 shadow-sm ${classes.border} ${classes.bg}`}
		>
			<h3 className={`text-base font-semibold ${classes.text}`}>
				{title}
			</h3>
			<div className="text-sm leading-6 text-muted-foreground [&_a]:text-primary [&_a]:underline [&_a]:underline-offset-4">
				{body}
			</div>
		</aside>
	);
}
```

`tone` 은 일반 문자열이므로, `Callout` 에는 `editMode`
분기가 없습니다 — 중립화할 라이브 링크나 포커스 가능한 컨트롤이
없기 때문입니다. 장식 전용 컴포넌트는
[editMode 패턴](/ko/guides/component-authoring/#4-the-editmode-pattern)의 가장 단순한 변형입니다.

### 7.3 설정 확정하기

`packages/components/src/callout/src/config.ts` 를 업데이트해
`defaultProps` 가 실제 예시를 고르고, `fields` 가 네 개의
톤을 `radio` 로 제공하도록 합니다:

```ts
import type {
	ComponentConfig,
	ComponentMetadata,
	Fields,
} from "@puckeditor/core";
import { createElement } from "react";
import packageJson from "../package.json";
import type { CalloutProps } from "./Callout";
import { Callout } from "./Callout";

export const metadata = {
	componentName: "Callout",
	componentSlug: "callout",
	packageName: packageJson.name,
	packageVersion: packageJson.version,
	scaffoldType: "content",
	schemaVersion: 1,
	suggestedCategory: "marketing",
} satisfies ComponentMetadata;

export const defaultProps = {
	title: "Heads up",
	body: "<p>Use this space for a helpful reminder.</p>",
	tone: "info",
} satisfies CalloutProps;

export const fields = {
	title: { type: "text", label: "Title" },
	body: { type: "richtext", label: "Body" },
	tone: {
		type: "radio",
		label: "Tone",
		options: [
			{ label: "Info", value: "info" },
			{ label: "Success", value: "success" },
			{ label: "Warning", value: "warning" },
			{ label: "Danger", value: "danger" },
		],
	},
} satisfies Fields<CalloutProps>;

const renderCallout: ComponentConfig<CalloutProps>["render"] = ({
	title,
	body,
	tone,
	editMode,
}) =>
	createElement(Callout, { title, body, tone, editMode });

export const calloutConfig = {
	label: "Callout",
	defaultProps,
	fields,
	metadata,
	render: renderCallout,
} satisfies ComponentConfig<CalloutProps>;

export const componentConfig = calloutConfig;
```

`src/index.ts` 는 생성된 그대로 유지하세요 — 이미 네 개의
정규 심볼과 `callout*` 카멜 별칭을 다시 내보냅니다.
