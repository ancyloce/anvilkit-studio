`@anvilkit/plugin-ai-copilot` は意図的にヘッドレスです。プロバイダーを
同梱せず、API クライアントをバンドルせず、あなたが
どのモデルを動かしているかも知りません。代わりに、ホストが所有する単一のコールバックを受け取ります。

```ts
createAiCopilotPlugin({
	generatePage: async (prompt, ctx) => {
		// you choose the LLM, you hold the secrets
		return somePageIR;
	},
	puckConfig,
});
```

このガイドでは、形状の選択が強制する信頼モデル、
正確な `generatePage()` コントラクト、schema コンテキストがどう構築されるか、すべてのレスポンスがディスパッチ前に通過する
バリデーター境界、エラーの
表示方法、コスト制御のつまみ、そして
実プロバイダーに対する 3 つの完全なアダプターを説明します。最後に明示的な「これをしてはいけない」セクションで締めくくります。

<Callout type="info" title="コードはグリーンを保つ">

	に抽出され、モック化された `fetch` を用いて CI 上で Vitest テストとして実行されます。もし
	コントラクトがドリフトすれば、ドキュメントが公開される前にビルドが失敗します。

## 1. 信頼モデル

ユーザーのプロンプトとレンダリングされたページの間には 3 つの主体が位置します。

```
┌─────────────────┐    ┌──────────────┐    ┌──────────────┐
│   Studio UI     │───▶│   your host  │───▶│    LLM       │
│ (this package)  │    │  (your app)  │    │ (Anthropic,  │
│                 │    │              │    │  OpenAI, …)  │
└─────────────────┘    └──────────────┘    └──────────────┘
        ▲                      │
        │     validated PageIR │
        └──────────────────────┘
```

**Studio はそれ自体ネットワークアクセスを持ちません。** あなたが
提供するコールバックを呼び出します。そのコールバックは認証情報が存在する唯一の場所であり、そのため
ホストのバックエンドが次の点を制御し続けます。

- どのプロバイダーが呼び出されるか。
- プロバイダーがどのモデルとパラメーターを受け取るか。
- そもそもどのユーザーが呼び出してよいか（認証、レート制限）。
- レスポンス本文がフィルタリングされないままクライアントに到達するかどうか。

コールバックは**関数参照**であり、設定
文字列ではないため、API キーをプラグインオプションに埋め込んで
誤ってシリアライズしてしまう方法はありません。サーバー
アクションから値を返す場合、シークレットはサーバー上に留まります。

<Callout type="warn" title="シークレットはホストバックエンドから決して出ない">
	完全な理由付けは
	[`docs/security/plugin-trust-model.md`](https://github.com/ancyloce/anvilkit-studio/blob/main/docs/security/plugin-trust-model.md)
	の §3 にあります。このガイドは、あなたが正式な
	能力境界をそこから読んでいることを前提とします。
</Callout>

## 2. `generatePage` コントラクト

```ts
import type {
	AiGenerationContext,
	PageIR,
} from "@anvilkit/core/types";

type GeneratePageFn = (
	prompt: string,
	ctx: AiGenerationContext,
) => Promise<PageIR>;
```

入力：

- `prompt`——生のユーザー文字列。受け取る際に
  サニタイズは一切行われません。プロバイダーが入力トークン単位で課金するなら、自分で長さを上限設定してください。
- `ctx.availableComponents: AiComponentSchema[]`——ホストの Puck `Config` の
  AI セーフな射影で、
  `@anvilkit/schema/configToAiContext()` によって構築されます。これは LLM
  が選択できる許可リストです。
- `ctx.currentData?`——プラグインに
  `forwardCurrentData: true` を渡したときにのみ設定されます。LLM が
  ページ上に既にあるブロックを参照できるようにします。

出力：**生の `PageIR` 候補**。有効である必要はありません
——プラグインはディスパッチ前にそれを `validateAiOutput()` に通します
（下記 §4）。きれいに失敗する方が、無理やり成功させるよりも望ましいです。

```ts
// minimum-viable generatePage (for tests or offline demos)
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

export const noopGeneratePage: GeneratePageFn = async (prompt, _ctx) => ({
	version: "1",
	root: {
		id: "root",
		type: "__root__",
		props: {},
		children: [
			{
				id: "hero-1",
				type: "Hero",
				props: { headline: prompt.slice(0, 80) },
			},
		],
	},
	assets: [],
	metadata: {},
});
```

## 3. Schema コンテキスト

`@anvilkit/schema` の `configToAiContext(puckConfig)` は、あなたが `<Studio>` に
渡すのと同じ Puck `Config` オブジェクトを走査し、ライブランタイムに
触れることなく LLM が推論できる
決定論的で JSON セーフなサマリーを生成します。

```ts
import { configToAiContext } from "@anvilkit/schema";
import type { AiComponentSchema } from "@anvilkit/core/types";

const ctx = configToAiContext(puckConfig);
// ctx.availableComponents: AiComponentSchema[]
//   [
//     {
//       componentName: "Hero",
//       description: "Marketing hero block with headline + body.",
//       fields: [{ name: "title", description: "…", type: "text" }]
//     },
//     { componentName: "Pricing", description: "…", fields: [...] },
//     ...
//   ]
```

2 つの保証が重要です。

1. **このリストは閉じた集合です。** LLM は
   `availableComponents` にないコンポーネント型を生み出すことはできません——バリデーターが
   それを拒否します（§4）。
2. **フィールドの種類は AI セーフです。** シリアライズ不可能なフィールド（slot、
   レンダラー）は `extractFieldSchema` によって除外されます。LLM は
   JSON としてラウンドトリップできるフィールドだけを見ます。

通常、`availableComponents` をそのままプロンプトに
渡すことはしません——選択したプロバイダーのツール／関数呼び出しの
慣習に合わせて整形してください（`@anvilkit/plugin-ai-copilot` はデータを提供します。それを
どう提示するかはプロバイダー固有です）。

## 4. バリデーター境界

すべての `generatePage` レスポンスは、プラグインが `puckApi.dispatch` を
呼び出す前に 1 つのゲートを通過します。

```ts
import { validateAiOutput } from "@anvilkit/validator";

const validation = validateAiOutput(response, ctx.availableComponents);
if (!validation.valid) {
	// emit error, do NOT dispatch
}
```

バリデーターは Zod schema です。次を強制します。

- `PageIR` の形状（`version === "1"`、ルートノード、assets 配列）。
- すべての子の `type` が `availableComponents` 内にあること。
- `props` がオブジェクトであること（シリアライズ可能性の制約は
  信頼モデルに記載されています。既知のギャップについてはそこの §4 を参照）。

次は強制**しません**。

- 文字列プロパティの内容（サニタイズはエクスポーターの仕事です）。
- サイズ制限（気にするなら自分の `generatePage` で強制してください——§6 を参照）。
- 意味上の妥当性（空の `headline` は構文上は有効です）。

失敗するとイベントバス上で `ai-copilot:error` を
`{ code: "VALIDATION_FAILED", issues }` とともに発行し、構造化ログ
行を書き込みます。その IR は破棄されます。部分的な状態が
コミットされることは決してありません。

## 5. エラーの表示

プラグインは 3 つの安定したエラーコードを発行します。

| コード              | いつ                                                   |
| ------------------- | ------------------------------------------------------ |
| `VALIDATION_FAILED` | レスポンスが Zod バリデーションに失敗した。            |
| `TIMEOUT`           | `generatePage` が `timeoutMs` 内に解決しなかった。     |
| `GENERATE_FAILED`   | `generatePage` がスローした（ネットワークエラー、クォータなど）。 |

あなたが保持する任意の `StudioPluginContext` 経由で購読します。

```ts
import type { StudioPluginContext } from "@anvilkit/core/types";
import type { AiCopilotErrorPayload } from "@anvilkit/plugin-ai-copilot";

function wireErrorToast(ctx: StudioPluginContext, toast: (msg: string) => void) {
	ctx.on("ai-copilot:error", (payload: AiCopilotErrorPayload) => {
		if (payload.code === "TIMEOUT") {
			toast("AI generation timed out. Try again.");
		} else if (payload.code === "VALIDATION_FAILED") {
			toast(`Response rejected: ${payload.issues?.[0]?.message ?? "bad IR"}`);
		} else {
			toast(`Generation failed: ${payload.message}`);
		}
	});
}
```

ロジックのために `payload.message` 文字列を解析しないでください——それらは
人間が表示するためのものです。`code` フィールドが安定したコントラクトです。

## 6. コスト制御

4 つのレバー。プラグインが提供する部分が多い順から、
あなた自身がやらなければならない部分が多い順へと降順に並べます。

1. **`timeoutMs`**（プラグインオプション、デフォルト 30 000 ミリ秒）。これを超えると、
   未解決の生成は中止され、`TIMEOUT` エラーが
   発火します。これはプロバイダーの病的なレイテンシを抑えます。
2. **`maxPromptLength`**（ホスト側、自分のアダプターで実装）。
   プラグインはプロンプト長を制限しません——暴走したプロンプトは LLM が
   応答する前にあなたのクォータに対して課金されます。あなたの `generatePage` 内の
   シンプルな `if (prompt.length > 4_000) throw` で
   十分です。
3. **レート制限**（ホスト側）。`generatePage` は
   ホストが所有する関数から呼ばれるため、ユーザー単位、IP 単位、
   組織単位でレート制限できます——バックエンドが持つどの認証コンテキストでも可能です。Studio 内には
   これをあなたの代わりに行えるものはありません。
4. **プロバイダー側のクォータ。** プロバイダーに渡す API キーが
   最後の防衛線です。支出上限付きのスコープ付きキーを使ってください。

## 7. 3 つの完全なアダプター

各アダプターは独立したパッケージとして `examples/ai-adapters/<provider>/`
の下に、それぞれの README とともに置かれています。このガイドでは
興味深い部分——`generatePage` の本体——をインラインで示し、ドリフトを CI
テストハーネスが捕捉できるようにします。

### Anthropic（Claude）

```ts
import Anthropic from "@anthropic-ai/sdk";
import { validateAiOutput } from "@anvilkit/validator";
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

const client = new Anthropic({ apiKey: process.env.AI_API_KEY });

export const generatePage: GeneratePageFn = async (prompt, ctx) => {
	if (prompt.length > 4_000) {
		throw new Error("Prompt too long");
	}
	const response = await client.messages.create({
		model: "claude-sonnet-4-6",
		max_tokens: 4_096,
		system:
			"You are a page generator. Return ONLY a JSON PageIR. " +
			"Allowed component types: " +
			ctx.availableComponents.map((c) => c.componentName).join(", "),
		messages: [{ role: "user", content: prompt }],
	});
	const block = response.content[0];
	if (block?.type !== "text") {
		throw new Error("No text block in Claude response");
	}
	return JSON.parse(block.text);
};
```

シークレット：`process.env.AI_API_KEY` はホストバックエンドに存在します。
`generatePage` 関数はサーバーモジュールからエクスポートされ、
サーバーコンポーネント内でのみホストの Studio 設定にインポートされます。

### OpenAI

```ts
import OpenAI from "openai";
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

const client = new OpenAI({ apiKey: process.env.AI_API_KEY });

export const generatePage: GeneratePageFn = async (prompt, ctx) => {
	if (prompt.length > 4_000) {
		throw new Error("Prompt too long");
	}
	const response = await client.chat.completions.create({
		model: "gpt-5",
		response_format: { type: "json_object" },
		messages: [
			{
				role: "system",
				content:
					"Return JSON PageIR only. Components: " +
					ctx.availableComponents.map((c) => c.componentName).join(", "),
			},
			{ role: "user", content: prompt },
		],
	});
	const text = response.choices[0]?.message?.content;
	if (!text) throw new Error("Empty OpenAI response");
	return JSON.parse(text);
};
```

### 汎用 `fetch`（プロバイダー非依存）

プロバイダーを自前のバックエンドルート経由でフロントに置く場合は、この形状を使ってください
（ブラウザ発のリクエストには推奨）。

```ts
import type { GeneratePageFn } from "@anvilkit/plugin-ai-copilot";

export function createFetchGeneratePage(endpoint: string): GeneratePageFn {
	return async (prompt, ctx) => {
		if (prompt.length > 4_000) {
			throw new Error("Prompt too long");
		}
		const response = await fetch(endpoint, {
			method: "POST",
			headers: { "content-type": "application/json" },
			body: JSON.stringify({
				prompt,
				availableComponents: ctx.availableComponents.map(
					(c) => c.componentName,
				),
			}),
		});
		if (!response.ok) {
			throw new Error(`AI endpoint returned ${response.status}`);
		}
		return (await response.json()) as Awaited<ReturnType<GeneratePageFn>>;
	};
}
```

その後、バックエンドの `POST /api/ai/generate-page` ルートが本物の
API キーを保持し、認証、レート制限、プロンプトサイズを一箇所で
強制できます。

## 8. これをしてはいけない

これらのアンチパターンは信頼モデルを破壊します——これらのいずれかを
含むビルドは決して出荷しないでください。

<Callout type="error" title="アンチパターン">
	**1. API キーをクライアント側でハードコードする。**

	```ts
	// ❌ NEVER
	const client = new Anthropic({
		apiKey: "sk-ant-xxxxxxxxxxxxxxxxxxxx",
	});
	```

	ページソースを見た者は誰でもキーを見ることができます。直ちにローテーションしてください。

	**2. ブラウザで動作するコード内で `import.meta.env.ANTHROPIC_API_KEY`
	からシークレットを読み取る。**

	バンドラーは `VITE_*`、`NEXT_PUBLIC_*` などの接頭辞を
	クライアントバンドルにインライン化します。環境変数がブラウザに出荷されるなら、それは
	クライアントシークレットです——漏洩したものとして扱ってください。

	**3. 認証情報を `studioConfig` に埋め込む。**

	`studioConfig` は登録されたすべてのプラグインから見えます（信頼
	モデル §2）。そこに置いたものは、ホストが後でインストールする
	サードパーティ製プラグインによって読み取り可能です。シークレットは
	サーバー専用の `generatePage` モジュールのモジュールスコープに保管してください。

	**4. `prompt` をサーバーから LLM へのパススルーとして信頼する。**

	プロンプトはユーザー入力です。プロンプトインジェクション攻撃は
	LLM を誘導して JS 形状のデータやホストが宣言していないコンポーネント
	型を返させようとするかもしれません。バリデーターは形状ドリフトを拒否しますが、
	アダプターは依然としてプロンプト長を上限設定し、
	明白な悪用を拒否すべきです（`validateAiOutput` §4 を参照）。

	**5. `generatePage` のエラーを黙って捕捉して握りつぶす。**

	スローさせてください。プラグインはスローされたエラーを
	`GENERATE_FAILED` イベントに変換し、UI がそれを表示できるようにします。エラーを
	握りつぶすと「生成をクリックしたのに何も起きなかった」が
	失敗モードになり、それは目に見える赤いメッセージよりもはるかに悪いです。
</Callout>
