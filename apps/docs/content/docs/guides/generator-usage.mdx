Every `@anvilkit/<slug>` component package starts from the same
generator: `pnpm -C packages/components gen:component`. It writes a
nine-file scaffold — `package.json`, `rslib.config.ts`, `tsconfig.json`,
`src/index.ts`, `src/<ComponentName>.tsx`, `src/config.ts`,
`src/styles.css`, `src/styles.d.ts`, and `README.md` — wired to the
workspace's Rslib, Tailwind, and Puck conventions. A fresh contributor
should be able to go from clone to a named, building package in under
five minutes.

This guide is the step-by-step walkthrough. For the contract the
scaffolded code must uphold, see [Component authoring](/guides/component-authoring/).

<Callout type="info" title="Skim order">
	If you already have the repo cloned, jump straight to §2 for the
	interactive run or §3 for the flag-driven run. §6 is a full
	reproducible worked example for a brand-new `@anvilkit/callout`
	package.
</Callout>

## 1. Prerequisites

The components workspace is a git submodule with its own pnpm
workspace. Miss the submodule init step and the generator will refuse
to run.

1. **Clone with submodules.**

	```bash
	git clone --recurse-submodules https://github.com/ancyloce/anvilkit-studio.git
	cd anvilkit-studio
	```

	If you already cloned without `--recurse-submodules`, run
	`git submodule update --init --recursive` once from the repo root.

2. **Pin the toolchain.** The repo enforces pnpm 10.33.0 via
	`packageManager` and targets Node ≥ 18.

	```bash
	corepack enable
	node -v        # ≥ 18
	pnpm -v        # 10.33.0
	```

3. **Install dependencies.** Run this from the repo root; Turbo and
	pnpm workspaces pick up `packages/components/` automatically.

	```bash
	pnpm install
	```

At this point `packages/components/src/` contains the existing
component packages (`button`, `hero`, `navbar`, …) and the generator
is ready to add one more.

## 2. Interactive run

The default invocation prompts for four answers, validates each, and
then writes the scaffold. Run it from the repo root so the `-C` flag
scopes the command to the components workspace:

```bash
pnpm -C packages/components gen:component
```

The prompts, in order:

| Prompt                  | Accepts                                                     | Default                         |
| ----------------------- | ----------------------------------------------------------- | ------------------------------- |
| **Component slug**      | lowercase kebab-case, unique under `src/`                   | _required_                      |
| **Display label**       | free-form string shown in the Puck rail                     | `Title-Cased Slug` (from slug)  |
| **Scaffold type**       | `content` · `layout` · `form`                               | _required_                      |
| **Suggested category**  | optional lowercase slug (`marketing`, `forms`, …)           | empty                           |

Validation rules (enforced by `turbo/generators/config.ts`):

- Slug must match `^[a-z0-9]+(?:-[a-z0-9]+)*$` — so `button-group` is
	fine, `ButtonGroup` and `my_button` are not.
- Slug must be unique — the generator refuses to overwrite
	`src/<slug>/` if it already exists.
- Category, if provided, must satisfy the same kebab-case shape.

A successful run prints one `>>> Added "…"` line per file and exits
clean.

## 3. Flag-driven run

Scriptable invocations skip the prompts. The `--` separator is
required so pnpm forwards the flags to the generator instead of
consuming them itself:

```bash
pnpm -C packages/components gen:component -- \
	--name button-group \
	--label "Button Group" \
	--template content \
	--category actions
```

Supported flags:

| Flag          | Required | Notes                                                       |
| ------------- | -------- | ----------------------------------------------------------- |
| `--name`      | yes      | Component slug; same validation as the interactive prompt.  |
| `--label`     | no       | Defaults to the slug in Title Case.                         |
| `--template`  | no       | Defaults to `content`. One of `content`, `layout`, `form`.  |
| `--category`  | no       | Written to `metadata.suggestedCategory` in `src/config.ts`. |

Invalid inputs fail fast with a message from the validator — no
scaffold is written, no partial state is left behind.

## 4. The three templates

Every template emits the same nine files, but the shape of
`<ComponentName>.tsx`, `config.ts`, and `package.json` differs so the
scaffold matches the component's intent. Pick the one that best fits
what you're building.

<Tabs items={['content', 'layout', 'form']}>

<Tab value="content">

**When to pick it.** Static marketing surfaces — buttons, heroes,
sections, pricing blocks. This is the default and the closest match
for most `@anvilkit/*` packages.

**What it scaffolds.**

- **Props:** `title: string`, `body: RichText`, `alignment?: 'left' | 'center' | 'right'`.
- **Fields:** `text` (title), `richtext` (body), `radio` (alignment).
- **`package.json` peer deps:** `@puckeditor/core`, `react`, `react-dom`.
- **No `@anvilkit/ui` dependency.**

**Render shape** (verbatim from `templates/component/content/src/component.tsx`):

```tsx
import type { RichText } from "@puckeditor/core";

export interface CalloutProps {
	title: string;
	body: RichText;
	alignment?: "left" | "center" | "right";
}

export interface CalloutViewProps extends CalloutProps {
	editMode?: boolean;
}
```

</Tab>

<Tab value="layout">

**When to pick it.** Structural containers that slot child components —
bento grids, two-column sections, page shells. Use this when the
component's job is to arrange other Puck components, not to render
terminal content itself.

**What it scaffolds.**

- **Props:** `title: string`, `leftColumn: Slot`, `rightColumn: Slot`, `emphasis?: 'balanced' | 'accent'`.
- **Fields:** `text` (title), `slot` × 2 (left/right columns), `radio` (emphasis).
- **Render:** the config's `render` wraps each `Slot` with
	`createElement(LeftColumn)` / `createElement(RightColumn)` so Puck
	mounts the nested drop zones.
- **`package.json` peer deps:** same as `content`.

The rendered component receives `leftColumn`/`rightColumn` as
`ReactNode` and threads `editMode` through — the scaffold already
wires the "Layout" affordance label that shows only while editing.

</Tab>

<Tab value="form">

**When to pick it.** Interactive inputs that need editor-aware
disabled state — text inputs, selects, form-field wrappers. This is
the only template that takes a runtime dependency on `@anvilkit/ui`.

**What it scaffolds.**

- **Props:** `label`, `name`, `type?` (text/email/password/search/tel/url), `placeholder?`, `helperText?`, `defaultValue?`, `required?`, `disabled?`.
- **Fields:** `text`, `text`, `select`, `text`, `textarea`, `text`, `radio`, `radio`.
- **`package.json` peer deps:** adds `@anvilkit/ui` alongside the usual Puck + React.
- **`styles.css`:** includes an extra `@source '../../../../ui/src/input.tsx'`
	so Tailwind picks up the upstream primitive's classes.
- **`editMode` handling:** the view component wraps `@anvilkit/ui`'s
	`Input` with `readOnly={editMode}` and a muted background so the
	field neutralizes itself in the editor.

</Tab>

</Tabs>

### Common scaffold files

Regardless of template, every scaffold includes:

- **`src/config.ts`** — exports `metadata`, `defaultProps`, `fields`,
	a `render<ComponentName>` adapter, `<slug>Config`, and `componentConfig`.
	`metadata.packageName` is sourced from `package.json` so rename-safe;
	`metadata.componentSlug` is written from the generator answer.
- **`src/index.ts`** — re-exports the component, its `Props` +
	`ViewProps` types, and the four canonical symbols (`componentConfig`,
	`defaultProps`, `fields`, `metadata`) under both names and the
	lowerCamel aliases.
- **`src/styles.css`** — starts with `@import '@anvilkit/tailwind-config/shadcn'`
	and the local `@source './**/*.{ts,tsx}'` directive.
- **`package.json`** — dual CJS/ESM `exports`, a `./styles.css`
	subpath, and an Rslib-driven `build` / `dev` / `typecheck` script set.

## 5. Post-scaffold checklist

The generator writes a working stub — it lints, typechecks, and
builds — but the stub is intentionally minimal. Finishing the package
follows the same five-step loop every time.

1. **Implement the render.** Open `src/<ComponentName>.tsx` and
	replace the placeholder markup with the real component. Keep the
	`<ComponentName>ViewProps` shape as the single source of truth for
	serializable props plus `editMode`.

2. **Finalize `defaultProps`, `fields`, and `metadata`.** In
	`src/config.ts`, tune the default values to something that looks
	intentional in the demo and refine the field schema — swap
	`radio`→`select` if the enum grows, add `array`/`object` fields
	for repeatable groups. Set `metadata.suggestedCategory` if it was
	empty.

3. **Wire peer deps + `@source` entries.** If you start consuming an
	`@anvilkit/ui` primitive, add it to `peerDependencies` +
	`devDependencies` and append its source path to `styles.css` as
	an extra `@source` directive.

4. **Validate.** The workspace has a single smoke command:

	```bash
	pnpm -C packages/components lint \
		&& pnpm -C packages/components typecheck \
		&& pnpm -C packages/components build
	```

	CI runs the same three, plus `madge` (circular deps), `publint`
	(package exports), and the Playwright suite.

5. **Integrate into the demo.** Two files to touch in `apps/demo/`:

	- **`apps/demo/lib/puck-demo.ts`** — add the import, the
		`DemoComponents` entry, the category wiring, the
		`demoConfig.components` entry, and an entry in `createDemoData`.
	- **`apps/demo/next.config.js`** — add `@anvilkit/<slug>` to the
		`transpilePackages` array so Next can transpile the source
		package.

	Run `pnpm -C apps/demo dev` and verify the component drops into
	the Puck editor at `/puck/editor`.

## 6. Publishing workflow

Versioning is Changesets-driven; every `@anvilkit/<slug>` releases
independently.

1. **Add a changeset.** From the components workspace:

	```bash
	pnpm -C packages/components changeset
	```

	Pick the new package, bump level (usually `minor` for a brand-new
	package), and write a user-facing description — it becomes the
	changelog entry verbatim.

2. **Open a PR.** Commit the changeset file alongside the component
	implementation. CI runs `lint`, `typecheck`, `madge`, `test`,
	`build`, `publint`, and the Playwright E2E suite on every push —
	see [`.github/workflows/ci.yml`](https://github.com/ancyloce/anvilkit-studio/blob/main/.github/workflows/ci.yml).

3. **Merge to `main`.** Changesets' GitHub action opens (or updates)
	a release PR that bumps the affected `package.json` versions and
	writes per-package `CHANGELOG.md` entries.

4. **Merge the release PR.** The action then runs:

	```bash
	pnpm -C packages/components release
	# = changeset version && pnpm build && changeset publish
	```

	Each package with a pending changeset publishes to npm at its new
	version. Consumers install only what they need —
	`pnpm add @anvilkit/<slug> @puckeditor/core`.

<Callout type="warn" title="Never hand-edit versions">
	Package versions live in `packages/components/src/<slug>/package.json`,
	but editing them directly bypasses Changesets' CHANGELOG generation
	and desyncs the release PR. Always route bumps through
	`pnpm changeset`.
</Callout>

## 7. Worked example — `@anvilkit/callout`

This walks through shipping a brand-new `callout` component from an
empty `src/callout/` directory to a demo integration. A fresh
contributor can reproduce it end-to-end in under 15 minutes.

### 7.1 Scaffold

From the repo root, after `pnpm install`:

```bash
pnpm -C packages/components gen:component -- \
	--name callout \
	--label "Callout" \
	--template content \
	--category marketing
```

After the run, `packages/components/src/callout/` contains the
nine-file scaffold, and the generated `src/config.ts` has
`metadata.suggestedCategory: 'marketing'` already wired.

### 7.2 Implement the render

Replace `packages/components/src/callout/src/Callout.tsx` with a
tone-aware container. The `tone` prop picks a shadcn semantic token
set so the component is theme-aware without inventing new colors:

```tsx
import type { RichText } from "@puckeditor/core";

export interface CalloutProps {
	title: string;
	body: RichText;
	tone?: "info" | "success" | "warning" | "danger";
}

export interface CalloutViewProps extends CalloutProps {
	editMode?: boolean;
}

const toneClasses: Record<
	NonNullable<CalloutProps["tone"]>,
	{ border: string; bg: string; text: string }
> = {
	info: {
		border: "border-border",
		bg: "bg-muted/40",
		text: "text-foreground",
	},
	success: {
		border: "border-primary/30",
		bg: "bg-primary/10",
		text: "text-primary",
	},
	warning: {
		border: "border-accent",
		bg: "bg-accent/20",
		text: "text-accent-foreground",
	},
	danger: {
		border: "border-destructive/40",
		bg: "bg-destructive/10",
		text: "text-destructive",
	},
};

export function Callout({
	title,
	body,
	tone = "info",
}: CalloutViewProps) {
	const classes = toneClasses[tone];

	return (
		<aside
			role="note"
			className={`grid gap-3 rounded-2xl border p-5 shadow-sm ${classes.border} ${classes.bg}`}
		>
			<h3 className={`text-base font-semibold ${classes.text}`}>
				{title}
			</h3>
			<div className="text-sm leading-6 text-muted-foreground [&_a]:text-primary [&_a]:underline [&_a]:underline-offset-4">
				{body}
			</div>
		</aside>
	);
}
```

Because `tone` is a plain string, the `Callout` has no `editMode`
branches — there are no live links or focusable controls to
neutralize. A decorative-only component is the simplest variant of
the [editMode pattern](/guides/component-authoring/#4-the-editmode-pattern).

### 7.3 Finalize the config

Update `packages/components/src/callout/src/config.ts` so
`defaultProps` picks a real example and `fields` offers the four
tones as a `radio`:

```ts
import type {
	ComponentConfig,
	ComponentMetadata,
	Fields,
} from "@puckeditor/core";
import { createElement } from "react";
import packageJson from "../package.json";
import type { CalloutProps } from "./Callout";
import { Callout } from "./Callout";

export const metadata = {
	componentName: "Callout",
	componentSlug: "callout",
	packageName: packageJson.name,
	packageVersion: packageJson.version,
	scaffoldType: "content",
	schemaVersion: 1,
	suggestedCategory: "marketing",
} satisfies ComponentMetadata;

export const defaultProps = {
	title: "Heads up",
	body: "<p>Use this space for a helpful reminder.</p>",
	tone: "info",
} satisfies CalloutProps;

export const fields = {
	title: { type: "text", label: "Title" },
	body: { type: "richtext", label: "Body" },
	tone: {
		type: "radio",
		label: "Tone",
		options: [
			{ label: "Info", value: "info" },
			{ label: "Success", value: "success" },
			{ label: "Warning", value: "warning" },
			{ label: "Danger", value: "danger" },
		],
	},
} satisfies Fields<CalloutProps>;

const renderCallout: ComponentConfig<CalloutProps>["render"] = ({
	title,
	body,
	tone,
	editMode,
}) =>
	createElement(Callout, { title, body, tone, editMode });

export const calloutConfig = {
	label: "Callout",
	defaultProps,
	fields,
	metadata,
	render: renderCallout,
} satisfies ComponentConfig<CalloutProps>;

export const componentConfig = calloutConfig;
```

Keep `src/index.ts` as generated — it already re-exports the four
canonical symbols plus the `callout*` camel aliases.

### 7.4 Validate

Run the three-command smoke from the components workspace:

```bash
pnpm -C packages/components lint \
	&& pnpm -C packages/components typecheck \
	&& pnpm -C packages/components build
```

All three should pass. `build` produces
`packages/components/src/callout/dist/` with `index.cjs`, `index.js`,
`index.d.ts`, and the compiled `styles.css`.

### 7.5 Integrate into the demo

Add the callout imports and wiring to `apps/demo/lib/puck-demo.ts`:

```ts
import {
	type CalloutProps,
	componentConfig as calloutComponentConfig,
	defaultProps as calloutDefaultProps,
} from "@anvilkit/callout";

export type DemoComponents = {
	// …existing entries…
	Callout: CalloutProps;
};

export const demoConfig: Config<DemoComponents> = {
	categories: {
		// …existing categories…
		marketing: {
			title: "Marketing",
			components: [
				// …existing marketing components…
				"Callout",
			],
		},
	},
	components: {
		// …existing components…
		Callout: calloutComponentConfig,
	},
};
```

Then add an entry to `createDemoData` so the component appears in the
seeded `/puck/render` payload:

```ts
{
	type: "Callout",
	props: {
		id: "callout-primary",
		...calloutDefaultProps,
},
},
```

And whitelist the package in `apps/demo/next.config.js`:

```js
transpilePackages: [
	// …existing packages…
	"@anvilkit/callout",
];
```

Start the demo and confirm Callout appears in the Marketing rail:

```bash
pnpm -C apps/demo dev
# Open http://localhost:3000/puck/editor
```

### 7.6 Ship it

Create the changeset, open the PR, and let the release PR publish
`@anvilkit/callout@0.1.0` once merged:

```bash
pnpm -C packages/components changeset
# Select @anvilkit/callout → minor
# Description: Add a new `callout` component package with four tones
# and shared demo integration.
```
