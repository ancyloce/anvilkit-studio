`StudioPlugin`은 `@anvilkit/core`의 표준 확장 지점입니다.
이 가이드는 계약의 모든 부분 —— meta, register
함수, 컨텍스트 API, 라이프사이클 훅, 헤더 액션, 내보내기
형식, dispatch 의미론, 테스트 —— 을 차례로 살펴보고,
실제로 npm에 게시 가능한 패키지로 제공되며 CI에서 실행되는 완성된 예제로 마무리합니다.

<Callout type="info" title="코드는 항상 그린 상태를 유지합니다">
	이 페이지의 모든 코드 블록은
	[`apps/docs/tests/guides/plugin-authoring.test.ts`](https://github.com/ancyloce/anvilkit-studio/blob/main/apps/docs/tests/guides/plugin-authoring.test.ts)
	로 추출되어 CI에서 Vitest 테스트로 실행됩니다. 계약이 어긋나면
	문서가 게시되기 전에 빌드가 실패합니다.
</Callout>

## 1. `StudioPlugin`이란

`@anvilkit/core`는 `createStudioConfig` 호출에서 혼합된
`plugins: (StudioPlugin | PuckPlugin)[]` 배열을 받습니다. 두 계약은 공존합니다.

| 기능                                           | `PuckPlugin` | `StudioPlugin` |
| ---------------------------------------------- | ------------ | -------------- |
| Puck의 rail / drawer / header 슬롯 오버라이드  | ✅           | ✅             |
| 데이터 변경 구독 / 게시 라이프사이클           | ❌           | ✅             |
| 내보내기 형식 제공(`html`, `react`, …)         | ❌           | ✅             |
| Studio 헤더 액션 제공                          | ❌           | ✅             |
| 병합된 `StudioConfig` 읽기                      | ❌           | ✅             |

`StudioPlugin`은 슈퍼셋입니다. Puck API를 통해 Puck 에디터와
대화하고, Puck 자체가 모델링하지 않는 횡단 관심사를 추가합니다.
Core는 `compilePlugins()` 동안 두 종류를 모두 해석하므로 호스트는
이들을 자유롭게 혼합할 수 있습니다.

## 2. 최소 동작 플러그인

가장 작은 합법적인 플러그인은 동결된 `meta` 블록과 빈 등록 결과를 반환하는
`register()` 함수입니다.

```ts
import type { StudioPlugin } from "@anvilkit/core/types";

export function createNoopPlugin(): StudioPlugin {
	const meta = {
		id: "com.example.noop",
		name: "No-op",
		version: "1.0.0",
		coreVersion: "^0.1.0-alpha",
	} as const;

	return {
		meta,
		register(_ctx) {
			return { meta };
		},
	};
}
```

이것만으로 `compilePlugins()`를 통과하기에 충분합니다. 추가되는 모든 기능
—— 훅, 오버라이드, 헤더 액션, 내보내기 형식 —— 은 모두 옵트인이며,
등록 객체의 별도 필드로 나타납니다.

<Callout type="warn" title="`coreVersion`은 계약입니다">
	`compilePlugins()`는 `meta.coreVersion`
	범위가 실행 중인 `CORE_VERSION`을 포함하지 않는 플러그인을 거부합니다. 오늘날 지원되는
	접두사는 정확 일치(`"0.1.0-alpha.0"`), 캐럿(`"^…"`),
	그리고 틸드(`"~…"`)입니다. 불일치는 마운트 시점에 `StudioPluginError`
	를 던지므로, 호스트는 어긋난 플러그인으로 렌더링하는 대신
	크게 실패할 수 있습니다.
</Callout>

## 3. `StudioPluginContext`

모든 라이프사이클 훅과 모든 헤더 액션의 `onClick`은
`StudioPluginContext`를 받습니다. 이것은 Core가 플러그인에게 건네주는
읽기 / 쓰기 / 관찰 핸들이며 —— 다른 어떤 주변 상태도 허용되지 않습니다.

| 필드           | 목적                                                                                     |
| -------------- | ---------------------------------------------------------------------------------------- |
| `getData()`    | 현재 Puck `Data`의 스냅샷(구독은 생성되지 않음).                                          |
| `getPuckApi()` | 라이브 `PuckApi`를 반환. 변경에는 `dispatch`를 사용; §6 참조.                             |
| `studioConfig` | 동결되고 병합된 `StudioConfig`(기본값 + 환경 변수 + 호스트 오버라이드).                   |
| `log(level, message, meta?)` | Core의 로거로 라우팅(`debug` / `info` / `warn` / `error`).                  |
| `emit(event, payload?)` | 프로세스 내 플러그인 이벤트 버스로 브로드캐스트(동기, 등록 순서로 전달).         |
| `on(event, handler)` | 버스 이벤트를 구독; 구독 해제 함수를 반환. `register`/`onInit`에서 구독.          |

두 가지 경계는 의도적입니다.

1. **store에 대한 직접 접근은 없습니다.** 플러그인은 Zustand store를 `import`하지 않습니다.
   모든 변경은 `getPuckApi().dispatch()`를 통해 흐르므로 Puck의
   히스토리 스택이 신뢰할 수 있는 단일 출처로 유지됩니다.
2. **`studioConfig`는 깊은 읽기 전용입니다.** 런타임에 이를 변경하는 것은 무동작이며,
   strict 모드에서는 던질 수 있습니다.

데이터 형태만 관찰하는 플러그인은 다음과 같이 구현할 수 있습니다.

```ts
import type {
	StudioPlugin,
	StudioPluginContext,
} from "@anvilkit/core/types";

export const observerPlugin: StudioPlugin = {
	meta: {
		id: "com.example.observer",
		name: "Observer",
		version: "1.0.0",
		coreVersion: "^0.1.0-alpha",
	},
	register(ctx: StudioPluginContext) {
		ctx.log("info", "observer plugin registered", {
			theme: ctx.studioConfig.theme.defaultMode,
		});
		return { meta: observerPlugin.meta };
	},
};
```

## 4. 라이프사이클 훅

훅은 `registration.hooks`에 위치합니다. 모든 훅은 선택적이며
`void` 또는 `Promise<void>`를 반환할 수 있습니다.

| 훅                | 발생 시점                                                             | 동시성       |
| ----------------- | -------------------------------------------------------------------- | ------------ |
| `onInit`          | 한 번, `compilePlugins()` 이후, `<Studio>`가 에디터를 마운트하기 전. | 병렬         |
| `onDataChange`    | 매 Puck `onChange`마다, 최신 `data` 인수와 함께.                     | 병렬         |
| `onBeforePublish` | 호스트가 게시하기 전 —— 중단하기 위해 `StudioPluginError`를 던질 수 있음. | **순차**     |
| `onAfterPublish`  | 게시가 성공한 후.                                                   | 병렬         |
| `onDestroy`       | `<Studio>`가 마운트 해제될 때.                                       | 병렬         |

두 가지 순서 규칙이 중요합니다.

1. **플러그인 순서.** 훅은 플러그인 등록 순서대로 —— 즉
   `createStudioConfig`에 전달된 `plugins: [...]` 배열의 순서대로 발생합니다.
2. **거부 의미론.** `onBeforePublish`는 **순차적으로** 실행됩니다. 처음으로
   던지는 훅이 나머지를 중단하고 라이프사이클 디스패처에서 다시 던집니다.
   다른 모든 이벤트는 `Promise.allSettled`를 통해 실행되므로,
   버그가 있는 플러그인은 로그만 남기고 결코 에디터를 충돌시키지 않습니다.

```ts
import type { StudioPlugin } from "@anvilkit/core/types";
import { StudioPluginError } from "@anvilkit/core";

export const requireHeroPlugin: StudioPlugin = {
	meta: {
		id: "com.example.require-hero",
		name: "Require Hero",
		version: "1.0.0",
		coreVersion: "^0.1.0-alpha",
	},
	register(_ctx) {
		return {
			meta: requireHeroPlugin.meta,
			hooks: {
				onBeforePublish(_ctx, data) {
					const hasHero = data.content.some((item) => item.type === "Hero");
					if (!hasHero) {
						throw new StudioPluginError(
							"com.example.require-hero",
							"Pages must include a Hero block before publish",
						);
					}
				},
			},
		};
	},
};
```

## 5. 내보내기 형식 제공

내보내기는 `registration.exportFormats`에 하나 이상의
`ExportFormatDefinition`을 제공합니다. 형식 `id`는 전역적으로 고유해야 합니다.
두 플러그인이 동일한 id를 등록하면 컴파일 시점에 `StudioPluginError`를 던집니다.

런타임은 `run()`에 정규화된 `PageIR`(Puck `Data`가 아님)을 건네주므로,
IR 정규화 단계는 모든 내보내기에 걸쳐 정확히 한 번만 발생합니다.

```ts
import type {
	ExportFormatDefinition,
	StudioPlugin,
} from "@anvilkit/core/types";

const jsonFormat: ExportFormatDefinition = {
	id: "json-snapshot",
	label: "JSON snapshot",
	extension: "json",
	mimeType: "application/json",
	async run(ir, _options) {
		return {
			content: JSON.stringify(ir, null, 2),
			filename: "page.json",
			warnings: [],
		};
	},
};

export function createJsonSnapshotPlugin(): StudioPlugin {
	const meta = {
		id: "com.example.json-snapshot",
		name: "JSON Snapshot Exporter",
		version: "1.0.0",
		coreVersion: "^0.1.0-alpha",
	} as const;

	return {
		meta,
		register(_ctx) {
			return { meta, exportFormats: [jsonFormat] };
		},
	};
}
```

프로덕션 참조 구현은
[`@anvilkit/plugin-export-html`](https://github.com/ancyloce/anvilkit-studio/tree/main/packages/plugins/plugin-export-html)
을 참조하세요 —— 이것은 에셋을 인라인하고, 컴포넌트별 CSS를 내보내며,
`@anvilkit/ir`을 통해 왕복 변환합니다.

## 6. 에디터 액션 디스패치

상태 변경은 항상 `getPuckApi().dispatch()`를 통해 흐릅니다. 플러그인이 가장 자주
사용하는 두 가지 패턴은 다음과 같습니다.

```ts
import type { StudioPluginContext } from "@anvilkit/core/types";

// Pattern A — full replacement (use sparingly; clobbers history).
export function clearPage(ctx: StudioPluginContext): void {
	ctx.getPuckApi().dispatch({
		type: "setData",
		data: { root: { props: {} }, content: [], zones: {} },
	});
}

// Pattern B — surgical action (preserves history; preferred).
export function insertHero(ctx: StudioPluginContext): void {
	ctx.getPuckApi().dispatch({
		type: "insert",
		componentType: "Hero",
		destinationZone: "default-zone",
		destinationIndex: 0,
	});
}
```

패턴 A는 단순하지만 트리 전체를 다시 쓰므로 실행 취소가
단일 단계로 접힙니다. 패턴 B는 Puck의 타입이 지정된 `PuckAction` 유니온
(`insert`, `move`, `remove`, `replace`, `duplicate`, …)을 사용하여 히스토리를
세밀하게 유지합니다. 새 트리가 이전 트리와 정말로 무관할 때(예: 저장된
초안을 여는 경우)에만 `setData`를 사용하세요 —— 그 외의 모든
경우에는 전용 액션이 있습니다.

## 7. 플러그인 테스트

플러그인 계약은 React가 없고 기본적으로 동기적이므로,
플러그인은 Node 환경에서 Vitest로 테스트할 수 있습니다. 표준
하네스는 다음과 같습니다.

1. `log`와 `emit`에 `vi.fn()`을 사용하고 `getPuckApi`를 스텁으로 둔 가짜
   `StudioPluginContext`를 구축합니다.
2. `compilePlugins([yourPlugin], ctx)`를 호출하여 `StudioRuntime`을 얻습니다.
3. `runtime.lifecycle.emit("onInit" | "onDataChange" | …, ctx, data)`를 구동합니다.
4. 플러그인이 기록한 것을 검증합니다 —— 카운트, 로그 호출, 디스패치된
   액션, `runtime.headerActions`에 제공된 헤더 액션.

```ts
import { compilePlugins, StudioConfigSchema } from "@anvilkit/core";
import type {
	StudioPlugin,
	StudioPluginContext,
} from "@anvilkit/core/types";
import type { Data as PuckData } from "@puckeditor/core";
import { describe, expect, it, vi } from "vitest";

const studioConfig = StudioConfigSchema.parse({});

function makeCtx(data: PuckData): StudioPluginContext {
	return {
		getData: () => data,
		getPuckApi: (() => {
			throw new Error("getPuckApi unused in this test");
		}) as unknown as StudioPluginContext["getPuckApi"],
		studioConfig,
		log: vi.fn(),
		emit: vi.fn(),
	};
}

const samplePlugin: StudioPlugin = {
	meta: {
		id: "com.example.sample",
		name: "Sample",
		version: "1.0.0",
		coreVersion: "^0.1.0-alpha",
	},
	register(_ctx) {
		return {
			meta: samplePlugin.meta,
			hooks: {
				onInit(ctx) {
					ctx.log("info", "sample plugin started");
				},
			},
		};
	},
};

describe("samplePlugin", () => {
	it("logs an info line on onInit", async () => {
		const ctx = makeCtx({ root: { props: {} }, content: [], zones: {} });
		const runtime = await compilePlugins([samplePlugin], ctx);

		await runtime.lifecycle.emit("onInit", ctx);

		expect(ctx.log).toHaveBeenCalledWith("info", "sample plugin started");
	});
});
```

그 단일 패턴만으로 React나 DOM을 건드리지 않고도
훅, 헤더 액션, 내보내기 형식, 그리고 이벤트 버스 발행을
연습하기에 충분합니다.

### 직접 만든 가짜보다 `@anvilkit/core/testing`을 선호하세요

Phase 4(`phase4-012`) 이후로, 동일한 가짜 컨텍스트가
재사용 가능한 픽스처로 내보내집니다. 모든
테스트 파일에서 `makeCtx`를 다시 작성하기보다 이를 선호하세요.

```ts
import {
	createFakeStudioContext,
	createFakePageIR,
	registerPlugin,
} from "@anvilkit/core/testing";

const harness = await registerPlugin(samplePlugin);
await harness.runInit();
expect(harness.ctx._mocks.logCalls[0]).toEqual([
	"info",
	"sample plugin started",
	undefined,
]);
```

`createFakeStudioContext()`는 `log` / `emit` /
`getPuckApi().dispatch`가 스파이로 뒷받침되는, 완전히 타입이 지정된
`StudioPluginContext`를 반환합니다. `_mocks.logCalls` / `_mocks.emitCalls` /
`_mocks.dispatchCalls`는 검증할 준비가 된 순서가 있는 배열입니다.
`createFakePageIR()`은 내보내기 /
validator 테스트를 위한 최소한의 유효한 PageIR을 구축합니다. `registerPlugin()`은 register +
라이프사이클 보일러플레이트를 래핑합니다. `@anvilkit/create-plugin` 스캐폴딩은 이러한 픽스처를 기본으로 사용합니다.

## 8. 완성된 예제 —— 컴포넌트 사용 횟수 카운터

패키지
[`@anvilkit/example-usage-counter`](https://github.com/ancyloce/anvilkit-studio/tree/main/examples/plugins/usage-counter)
는 실제로 npm에 게시 가능한 워크스페이스 패키지로 제공됩니다. 이것은
매 Puck `onChange`마다 `componentType → instanceCount`를 추적하고, 합계를
세 개의 독립적인 표면을 통해 노출하며, 다섯 개의 플러그인 기여 슬롯 중 네 개를
100줄 미만으로 시연합니다.

### 해부

플러그인 팩토리:

```ts
import type { Data as PuckData } from "@puckeditor/core";
import type {
	StudioHeaderAction,
	StudioPluginContext,
	StudioPluginMeta,
} from "@anvilkit/core/types";

const META: StudioPluginMeta = {
	id: "anvilkit-example-usage-counter",
	name: "Component Usage Counter",
	version: "0.1.0",
	coreVersion: "^0.1.0-alpha",
};

function computeCounts(data: PuckData): Readonly<Record<string, number>> {
	const counts: Record<string, number> = {};
	for (const item of data.content) {
		counts[item.type] = (counts[item.type] ?? 0) + 1;
	}
	const zones = data.zones ?? {};
	for (const zoneKey of Object.keys(zones)) {
		for (const item of zones[zoneKey] ?? []) {
			counts[item.type] = (counts[item.type] ?? 0) + 1;
		}
	}
	return counts;
}

export function createUsageCounterPluginCore() {
	let counts: Readonly<Record<string, number>> = {};

	function update(next: typeof counts, ctx: StudioPluginContext): void {
		counts = next;
		ctx.emit("usage-counter:update", counts);
	}

	const action: StudioHeaderAction = {
		id: "usage-counter-log",
		label: "Log usage counts",
		icon: "list",
		group: "overflow",
		onClick(ctx) {
			ctx.log("info", "usage-counter:snapshot", { counts });
		},
	};

	return {
		meta: META,
		getCounts: () => counts,
		register(_ctx: StudioPluginContext) {
			return {
				meta: META,
				hooks: {
					onInit: (ctx: StudioPluginContext) =>
						update(computeCounts(ctx.getData()), ctx),
					onDataChange: (ctx: StudioPluginContext, data: PuckData) =>
						update(computeCounts(data), ctx),
				},
				headerActions: [action],
			};
		},
	};
}
```

### 호스트에서 연결하기

```ts
import { createUsageCounterPlugin } from "@anvilkit/example-usage-counter";
// In the host:
//   const usageCounter = createUsageCounterPlugin({ verbose: true });
//   const config = createStudioConfig({ plugins: [usageCounter] });
//   usageCounter.subscribe((counts) => console.log(counts));
```

`createUsageCounterPlugin`은 표준 `StudioPlugin` 형태에 더해,
두 개의 추가 메서드(`getCounts()`와 `subscribe(fn)`)를 반환합니다. 이는 이벤트 버스를
구독하지 않고도 라이브 카운트를 읽고자 하는 호스트와 테스트를 위한 것입니다.

### 테스트하기

전체 테스트 파일은

## 9. 플러그인 설정 및 타입 안전성

### 플러그인 타입은 `<Studio>`로 전파되지 않습니다

이것은 우회해야 할 빈틈이 아니라 **의도적인 아키텍처
결정**입니다. `<Studio>`는 그 플러그인 집합에 대해 제네릭이 *아닙니다*.
`plugins={[myPlugin]}`을 전달해도 `myPlugin`의 설정 타입이
`config`로 흐르지 않으며, `config.experimental`은 `Record<string, unknown>` 타입입니다.

이유: core는 자신의 타입 버전을 모든 플러그인의 버전과 분리합니다. 플러그인은
`@anvilkit/core`와 독립적으로 새 메이저 버전을 출시할 수 있으며, 어떤 플러그인
업그레이드도 결코 core의 타입 버전 인상을 강제하지 않습니다. 그 대가로 호스트의
`<Studio>` 경계는 **플러그인에 특화된 설정에 대해 컴파일 타임 검증이나 IDE
자동 완성을 전혀 제공하지 않습니다** —— 그러나 플러그인 자신의
`register()` 본문 내부에서는 `StudioPluginContext<UserConfig>`가 완전히 타입이 지정되므로,
플러그인 작성자는 플러그인 코드를 작성하는 곳에서 완전한 안전성을 유지합니다.

**플러그인 설정은 `<Studio>` 타입 경계가 아니라 Zod를 통해 런타임에 검증됩니다**.
`createStudioConfig()`는 `StudioConfigSchema`(`config.experimental`은
`z.record(z.string(), z.unknown())`)를 실행하므로, 잘못된 형식의 설정은 시작 시점에
Zod 오류와 함께 빠르게 실패합니다 —— 다만 `tsc` 시점은 아닙니다.

### 플러그인의 `experimental` 설정 타입을 직접 지정하기

Core는 모듈 보강 슬롯 `StudioExperimentalConfig`를 노출합니다.
선언 병합으로 키를 추가하면, core가 당신의 타입에 의존하지 않고도
*당신의* 플러그인 설정에 대한 인텔리센스 + 검사를 얻을 수 있습니다.

```ts
declare module "@anvilkit/core" {
  interface StudioExperimentalConfig {
    myPlugin?: { apiKey: string; debug?: boolean };
  }
}

// Now this is type-checked and autocompleted:
const config = createStudioConfig({
  experimental: { myPlugin: { apiKey: "…" } },
});
```

이 슬롯은 열린 `Record<string, unknown>`과 교차하므로, 순전히
가산적입니다 —— 보강되지 않은 키는 타입 없이 계속 작동하며,
기존 설정은 깨지지 않습니다. 어쨌든 런타임은 여전히 Zod를 통해 검증합니다.

### 함정: 함수를 포함하는 `config`를 메모이즈하기

`<Studio>`는 플러그인을 언제 다시 컴파일할지 결정하기 위해 `config`의
지문을 계산합니다. 순수 JSON 설정은 저렴한 구조적 해시를 사용합니다. 그러나 `config`가
**함수, symbol, 또는 bigint**를 포함하면(예:
`config={{ experimental: { transform: fn } }}`), 구조적 해싱은
그것을 조용히 누락시키므로, core는 안전하게 **참조
동일성**으로 폴백합니다. 그 결과로, *인라인* 비 JSON 설정 객체는
매 렌더링마다 새로운 동일성을 얻게 되어, 부모가 렌더링될 때마다
전체 플러그인 컴파일이 다시 실행됩니다.

해결: 비 JSON 값을 담은 설정은 모두 메모이즈(또는 호이스트)하세요.

```tsx
// ❌ recompiles every render
<Studio config={{ experimental: { transform: (n) => n } }} />

// ✅ stable identity → compiles once
const config = useMemo(
  () => ({ experimental: { transform: (n) => n } }),
  [],
);
<Studio config={config} />
```

개발 환경에서는, core가 이 패턴(동일한 구조적 설정이 연속된 렌더링에서
새로운 동일성과 함께 도착하는 것)을 감지하면 일회성
`console.warn`을 기록합니다.
