## Classes

| Class | Description |
| ------ | ------ |
| [DebouncedAdapterDestroyedError](classes/DebouncedAdapterDestroyedError.md) | Wrap a `SnapshotAdapter` so that bursts of `save()` calls within `ms` coalesce into a single underlying write. The latest IR wins; every caller that issued a save during the window receives the resolved id (or the rejection) of that single flush. |
| [SnapshotCorruptedError](classes/SnapshotCorruptedError.md) | The record is internally inconsistent â€” the payload failed to decode (corrupt / schema-drifted), or metadata is present without its payload (which, because retention deletes both atomically in one synchronous transaction, indicates corruption, not eviction). |
| [SnapshotNotFoundError](classes/SnapshotNotFoundError.md) | No snapshot with this id exists in the shared Y.Doc. |
| [SnapshotPrunedError](classes/SnapshotPrunedError.md) | The snapshot was evicted by the `maxSnapshots` retention cap. This is best-effort and **same-session**: it is raised when THIS adapter instance performed the eviction (it tracks its own pruned ids). Cross-peer or post-reload, an evicted id is indistinguishable from one that never existed (retention deletes meta+payload atomically) and surfaces as [SnapshotNotFoundError](classes/SnapshotNotFoundError.md); a host history UI should pre-filter requests against the retained `list()`. Recoverable: hide / disable the revision rather than treat it as corruption. |
